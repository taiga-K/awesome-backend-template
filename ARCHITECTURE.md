# Architecture

この文書は、このリポジトリの高レベルなアーキテクチャを説明する。
目的は、プロジェクトの全体像とコードマップを短く共有し、新しく読む人が
「どこに何があるか」「どこを直すべきか」を判断しやすくすることにある。

この文書は実装の詳細仕様ではない。変わりやすい関数名、個別の API 一覧、
一時的な実装都合はここに書かない。代わりに、変わりにくい責務分担、
依存方向、文書の置き場、境界条件を記述する。

ファイルやモジュールはリンクせず、名称で検索できる前提で記す。

## 全体像

このリポジトリは、Go を前提にした DDD x クリーンアーキテクチャの
バックエンドアプリケーションとして構成する。

関心は大きく次の 3 つに分かれる。

- アプリケーション本体のコード
- ドメイン知識と外部契約を記述する文書
- 実装と文書を支える設定や検証ルール

高レベルには、外部からの入力は `cmd/api` を起点に
`internal/presentation` から入り、`internal/usecase` が処理の流れを調停し、
`internal/domain` が業務ルールを表現する。永続化や外部サービス接続は
`internal/infra` に閉じ込める。設定の読み込みや組み立ては
`internal/config` と `cmd/api` が担う。

業務用語やユースケースは `docs/domain` に置き、HTTP の外部契約は
`docs/api` に置く。つまり、コードだけでなく文書にも境界がある。

```text
Client / External System
        |
        v
      cmd/api
        |
        v
internal/presentation
        |
        v
 internal/usecase
        |
        v
  internal/domain
        ^
        |
  internal/infra

docs/domain: 業務の言葉とユースケース
docs/api: HTTP の契約
```

## Code Map

### ルート

リポジトリのルートには、実装コード、設計文書、開発ルールが並ぶ。

- `cmd/api`: アプリケーションの起動点。依存関係を組み立て、外側の世界と接続する。
- `internal`: アプリケーション本体。レイヤーごとの責務をここで分ける。
- `pkg`: アプリケーション内部事情に依存しない、再利用可能なパッケージの置き場。
- `docs/domain`: 業務の言葉、ユースケース、モデル理解のための文書を置く。
- `docs/api`: OpenAPI を中心とした外部 API 契約を置く。
- `README.md`: プロジェクトの入口と利用者向け概要を置く。
- `CONTRIBUTING.md`: 開発手順と変更時の共通ルールを置く。

### `internal`

`internal` はアプリケーションの中心であり、レイヤーごとに責務を分ける。

- `internal/domain`: ドメインルール、集約、エンティティ、値オブジェクト、ドメインサービス、リポジトリ抽象など、業務の中核を置く。
- `internal/usecase`: ユースケース単位の処理の流れ、認可、トランザクション境界、副作用の順序づけ、外部ポートの呼び分けを置く。
- `internal/presentation`: HTTP など外部入出力との接点を置く。request/response 変換、認証情報の橋渡し、エラー表現の変換を担う。
- `internal/infra`: DB、ORM、SQL、外部 API、SDK、メッセージングなど、実装詳細を置く。`domain` や `usecase` が要求する抽象の実装はここに置く。
- `internal/config`: 設定の読み込み、構成値の解釈、外側の依存を初期化するための設定を置く。

### `docs/domain`

`docs/domain` は業務理解の入口であり、コードの外にあるがアーキテクチャの内側に属する。

- `ubiquitous-language.md`: 用語の意味、境界、言い換えを置く。実装詳細や API パラメータ一覧の置き場ではない。
- `usecase`: アクター、目的、前提、結果、主な流れを置く。
- `uml`: モデルの関係や構造を補助的に表現する。

コードから業務意図を読み取りにくいときは、まず `docs/domain` を見る。

### `docs/api`

`docs/api` は外部公開する HTTP 契約の置き場である。

- `openapi.yaml`: OpenAPI のルート定義を置く。
- `paths`: エンドポイント単位の path 定義を置く。
- `components/schemas`: 外部に見せるデータ構造を置く。

このリポジトリでは、OpenAPI は最初から分割構成で扱う前提を持つ。
`docs/api` はプレゼンテーション層の契約面を表すが、業務用語の正本そのものではない。

## どこを見るか

何を知りたいかによって、最初に見る場所は変わる。

- 業務用語や意味を知りたいとき: `docs/domain/ubiquitous-language.md`
- ユースケースの流れを知りたいとき: `docs/domain/usecase`
- どのルールが業務の中核かを知りたいとき: `internal/domain`
- リクエストからユースケースへの入り口を知りたいとき: `internal/presentation`
- 複数のドメインオブジェクトや外部接続をまたぐ処理を知りたいとき: `internal/usecase`
- DB や外部サービスの具体的な接続方法を知りたいとき: `internal/infra`
- アプリ全体の組み立て方を知りたいとき: `cmd/api` と `internal/config`
- HTTP 契約を知りたいとき: `docs/api`

## アーキテクチャ上の不変条件

このリポジトリでは、ソースコードを読むだけでは分かりにくい不変条件を重視する。

- 依存方向は外側から内側へ向く。`presentation` と `infra` は `usecase` や `domain` を利用してよいが、`domain` は `presentation` や `infra` を知らない。
- `domain` は業務ルールの置き場であり、HTTP、JSON、ORM、SQL、SDK の都合を持ち込まない。
- `usecase` は調停役であり、業務知識の本体そのものではない。単一集約の不変条件はまず `domain` に置く。
- `presentation` は外部入出力とアプリケーションの橋渡しを担う。Web フレームワーク依存、認証方式、レスポンス整形はここに閉じ込める。
- `infra` は実装詳細の置き場であり、DB row や SDK の型を内側へ漏らさない。
- `cmd/api` は組み立ての場所であり、業務ロジックの本体を置く場所ではない。
- `docs/domain` は業務理解の入口であり、`docs/api` は外部契約の入口である。両者は役割が異なる。
- `pkg` は本当に独立して再利用したいものだけを置く。アプリケーション固有の中心ロジックは `internal` に残す。
- `shared`、`common`、`util` のような曖昧な退避先を安易に増やさない。責務名で分ける。

## 境界

境界は、何が存在するかだけでなく、何を置かないかでも定義する。

### Domain Boundary

`internal/domain` には業務の判断を置く。
ここには HTTP の入出力形式、永続化方式、フレームワークの型を置かない。

### Usecase Boundary

`internal/usecase` には処理順序と調停を置く。
ここには SQL 文、HTTP レスポンスの整形、ORM モデルを置かない。

### Presentation Boundary

`internal/presentation` には外部とのインターフェースを置く。
ここには業務ルールの本体や永続化の詳細を置かない。

### Infrastructure Boundary

`internal/infra` には外部技術との接続を置く。
ここには集約の不変条件や API 契約の本体を置かない。

### Documentation Boundary

`docs/domain` は「業務の言葉と意図」を記述する。
`docs/api` は「外部に見せる契約」を記述する。
コード変更時に両方を同じ粒度で更新するとは限らないが、意味や挙動が変わるなら整合は取る。

## 横断的な関心事

設計全体を通して重視するのは、高凝集・低結合である。
ひとつのモジュールはひとつの責務に集中し、モジュール間は必要最小限の
インターフェースで接続する。

また、このリポジトリではコードだけでなく文書も成果物として扱う。
利用方法、挙動、用語、契約に影響する変更は、対応する文書も更新対象になる。

## この文書の見方

この文書は「いま何が実装済みか」を列挙するものではなく、
「実装をどこに置くべきか」「どの境界を崩さないべきか」を示す地図として読むのがよい。
