# Contributing Guide
この文書は、現在の個人開発と将来のチーム開発の両方で使える、開発参加時の基本ルールをまとめたものです。

## 目的

- 開発の進め方を揃える
- レビューしやすい変更単位を保つ
- 将来の参加者が迷わず開発を始められるようにする

## 開発方針

- 本プロジェクトの開発言語は Go を前提とします。
- 変更は小さく保ち、1つの Pull Request で扱う関心事を絞ってください。
- 仕様や挙動に影響する変更を行った場合は、コードだけでなく関連ドキュメントも更新してください。
- 特に `README.md` の更新は必須です。利用方法、前提条件、挙動、セットアップ手順に影響がある場合は必ず反映してください。

## ブランチ運用

- ブランチ運用は GitFlow を採用します。
- 作業は適切な作業ブランチで行い、`main` へ直接コミットしないでください。
- ブランチ名は GitFlow の考え方に沿って、変更の意図が分かる名前を付けてください。

## コミットメッセージ

- コミットメッセージは Conventional Commits を採用します。
- `type` は Conventional Commits に従って英語で記述してください。
- `subject` と `body` は日本語で記述してください。
- 1つのコミットには、できるだけ1つの意図を持たせてください。

例:

```text
feat: Slack メッセージ集計の初期実装を追加

日別の投稿数を集計する処理を追加した。
後続で可視化画面から参照できるようにする。
```

## Pull Request

- 変更は Pull Request ベースで進めます。
- Pull Request 作成時は `.github/PULL_REQUEST_TEMPLATE.md` を使用してください。
- Pull Request には、変更概要、テスト方法、影響範囲、レビューしてほしい観点を明記してください。
- 大きな変更は、実装前または早い段階で方針を共有してください。

# レビューしやすい変更

- 無関係な修正を同じ Pull Request に含めないでください。
- リネーム、整形、機能追加、リファクタリングは可能な限り分けてください。
- 自明でない設計判断がある場合は、Pull Request の説明文に理由を書いてください。

## ドキュメント更新

- 実装変更により利用方法や挙動が変わる場合は `README.md` を必ず更新してください。

## 不明点がある場合

- 仕様が曖昧なまま実装を進めず、先に方針を確認してください。
- 文書にない運用ルールが必要になった場合は、その場限りにせずドキュメントへ反映することを検討してください。
