# Coding Style

## イミュータビリティ（重要）

常に新しいオブジェクトを作成し、既存のものを変更してはいけない：

```
// 擬似コード
誤り:  modify(original, field, value) → 元のオブジェクトをその場で変更
正解:  update(original, field, value) → 変更を反映した新しいコピーを返す
```

理由：イミュータブルなデータは隠れた副作用を防ぎ、デバッグを容易にし、安全な並行処理を可能にする。

## ファイル構成

少数の大きなファイルよりも多数の小さなファイルを推奨：
- 高凝集、低結合
- 200〜400行が目安、最大800行
- 大きいモジュールからユーティリティを抽出する
- 型ではなく機能/ドメインで整理する

## エラーハンドリング

エラーは常に網羅的に扱う：
- すべてのレベルで明示的にエラーを処理する
- UI向けコードではユーザーフレンドリーなエラーメッセージを提供する
- サーバー側では詳細なエラーコンテキストをログに記録する
- エラーを黙って飲み込まない

## 入力検証

システム境界で常に検証する：
- 処理の前にすべてのユーザー入力を検証する
- 可能な場合はスキーマベースの検証を使用する
- 明確なエラーメッセージとともに早めに失敗する
- 外部データ（APIレスポンス、ユーザー入力、ファイル内容）を信用しない

## コード品質チェックリスト

作業完了前に確認すること：
- [ ] コードが読みやすく、適切に命名されている
- [ ] 関数が小さい（50行未満）
- [ ] ファイルが目的に絞られている（800行未満）
- [ ] 深いネストがない（4段階以下）
- [ ] 適切なエラーハンドリングがある
- [ ] ハードコードされた値がない（定数や設定を使用）
- [ ] ミューテーションがない（イミュータブルなパターンを使用）
