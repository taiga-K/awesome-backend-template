# 種別ごとの出力ルール

## 起動ログ

- レベルは `INFO`
- まず人が見て判断しやすい情報を出す
- 少なくとも以下を含める
  - `service.name`
  - `service.version`
  - Git ハッシュ
  - ビルド時刻
  - `deployment.environment`
  - 主要設定値の要約
- 主要設定値に秘密情報を含めない
- 起動シーケンスのマイルストーンを残す

## Web API アクセスログ

- 可能ならリクエスト時とレスポンス時の両方を出す
- 少なくともレスポンス時のログは出し、処理結果と所要時間を残す
- ミドルウェアで一元化する
- ヘルスチェックは除外する
- 推奨項目
  - `timestamp`
  - `request.id`
  - `trace_id`
  - `url.path`
  - `http.request.method`
  - `http.response.status_code`
  - `http.server.request.duration`
  - `user_agent.original`
  - `client.address` が許可される場合はそれも含める

## バッチログ

- 開始と終了は必ず `INFO` で出す
- `batch.job_id`、入力パラメータ、結果件数、処理時間を残す
- 大量処理は進捗をサンプリングして出す
- ループ内で 1 件ごとに出さない
- 0 件スキップも、運用価値があるなら `INFO` で出す

## 例外ログ

- ログに残すのは、処理責務上その場で意味がある地点だけにする
- 同じ例外を何層も重複出力しない
- `exception.type`、`exception.message`、必要なら `exception.stack_trace` を持たせる
- ユーザーへ返す文言は別管理にし、必要なら `trace_id` で紐づける

## ローカル開発時

- 非構造ログ、色付け、絵文字はローカル限定なら可
- デプロイメント環境へ持ち込む設定にしない
- CI やログ基盤でノイズになる ANSI エスケープは避ける

## セキュリティ

- 出してはいけないもの
  - パスワード
  - セッション ID
  - アクセストークン
  - API キー
  - クレジットカード番号
  - 不要な PII
- 危険な値を含む構造体をそのまま `slog.Any()` しない
- ログ閲覧権限と改ざん防止は別途インフラで担保する

## 性能と費用

- `slog` では属性渡しを優先し、不要な文字列生成を避ける
- 出力しないレベルのために重い整形処理をしない
- SQL や巨大ペイロードの大量出力を避ける
- 本当に必要なログだけを残し、保持期間も環境別に決める
