# レビュー観点

## 設計確認

- 対象ログの読者と利用目的が明確か
- 共通項目と拡張項目の境界が明確か
- キー名が `dot.case` で統一されているか
- アプリコードがベンダ固有キーへ依存していないか

## メッセージ確認

- `message` が事実だけを簡潔に表しているか
- 可変値が属性へ分離されているか
- 調査に必要な識別子が十分か
- ユーザー向け文言と混在していないか

## ログレベル確認

- `DEBUG`、`INFO`、`WARN`、`ERROR` の使い分けが妥当か
- 本番で不要な `DEBUG` が常時出ないか
- 通知はログレベルだけで足りるか
- メッセージコードが必要なログにだけ付いているか

## 種別別確認

- Web API はミドルウェアでアクセスログを一元化できているか
- ヘルスチェックを除外しているか
- バッチは開始、終了、件数、進捗が追えるか
- 起動ログにサービス名、バージョン、環境があるか

## セキュリティ確認

- 秘密情報や PII が出力されていないか
- 構造体丸ごと出力で危険な値が漏れていないか
- IP やユーザー ID の出力が要件と整合しているか
- スタックトレースをユーザーへ返していないか

## 性能と費用確認

- ループ内で高頻度ログを出していないか
- 巨大ペイロードや全文 SQL を無制限に出していないか
- 文字列連結や重い整形を毎回行っていないか
- 保持期間と検索頻度に見合う量か

## Go / `slog` 確認

- `logger.InfoContext()` などに属性で値を渡しているか
- `context.Context` から `trace_id` などを引き継げるか
- 共通属性の付与をアプリ各所で重複実装していないか
- `source` やスタック取得の高コスト項目を常時有効にしていないか
