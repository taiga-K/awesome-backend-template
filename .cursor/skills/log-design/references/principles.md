# 原則と決め方

## スコープ

- 対象はアプリケーションが出力するログ
- 主眼はバックエンドのアプリログ、アクセスログ、バッチログ、起動ログ
- クラウドやインフラが自動出力する監査ログやフローログは対象外

## このリポジトリの前提

- 実行環境はクラウド前提
- デプロイメント環境ではファイル出力ではなく `stdout` を基本にする
- バックエンドは Go + Gin + `slog`

## 優先順位

1. 調査可能性: 障害時に「何が起きたか」を追えること
2. 一貫性: 同じ種類の事象が同じキー名と粒度で出ること
3. セキュリティ: 機密情報や不要な個人情報を出さないこと
4. 費用: 低価値ログで保管費・検索費を増やさないこと
5. 可搬性: アプリコードが特定ベンダのログ仕様に縛られないこと

## キー名の考え方

- 命名は OpenTelemetry のセマンティック規約を優先する
- 足りない語彙は ECS から補う
- ただし Go の `slog` 既定出力で `time`、`level`、`msg` が出ることは許容する
- 収集基盤や共通ハンドラで `timestamp`、`severity.text`、`message` 相当へ正規化してもよい
- アプリケーションごとに独自キーを増やしすぎない

## ログレベル

- 業務コードでは基本的に `DEBUG` から `ERROR` を使う
- 標準の Go `slog` には `TRACE` がないため、業務コードでは基本的に `DEBUG` から `ERROR` を使い、`TRACE` や `FATAL` 相当は共通基盤や独自ラッパー側に寄せる
- 環境別の標準は以下を基本にする
  - local: 標準 `slog` 前提では `DEBUG` まで許容
  - dev: `DEBUG` まで許容
  - stg / prod: `INFO` 以上を基本
- 切り替えは環境変数で行い、実行中に外部 API で変更する設計は避ける

## メッセージの書き方

- `message` は事実を一意に解釈できる短文にする
- 可変値は文字列連結せず、構造化属性に分離する
- 調査に必要な識別子を含める
  - `trace_id`
  - `request.id`
  - `user.id`
  - `tenant.id`
  - 業務 ID 例: `order.id`, `product.id`
- ユーザー向けメッセージとログメッセージは分離する

## メッセージコードと通知

- 運用者対応が必要な `WARN` 以上には、必要に応じて一意なメッセージコードを付与する
- まずはログレベルだけで通知を制御できないか検討する
- 通知フラグは、`INFO` でも通知したいなど明確な理由がある場合のみ追加する

## 自動付与と明示出力

- 共通処理で自動付与したいもの
  - タイムスタンプ
  - ログレベル
  - トレース ID / リクエスト ID
  - HTTP メソッド、パス、ステータス、処理時間
- 業務コードで明示出力するもの
  - 業務イベントの事実
  - ドメイン識別子
  - メッセージコード
  - 例外や失敗理由の補足

## ローカル環境

- ローカル専用なら非構造ログや色付けを許容してよい
- ただしデプロイメント環境へそのまま持ち込まない
- ANSI エスケープや絵文字は本番ログでは使わない
