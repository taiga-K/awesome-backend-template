# バージョニング・運用・セキュリティ

## バージョニング

- バージョン更新は最小限にする
- 後方互換性を保てる変更は、なるべく既存バージョン内で吸収する
- 破壊的変更が避けられないときだけメジャーバージョンを上げる
- URI にメジャーバージョンを含めるのを基本とする
- 例: `/v1/tasks`

## 互換性の扱い

- フィールド追加は互換的変更として扱えるかを検討する
- フィールド削除や意味変更は破壊的変更として扱う
- 最新版への暗黙 alias は避ける

## 廃止

- 廃止は事前告知する
- 移行期間を設ける
- 提供終了時は `410 Gone` を検討する

## 認証と認可

- 外部公開や複数クライアント連携では OAuth 2.0 を優先候補にする
- スコープ設計を行う
- 認証方式は API 全体で統一する

## HTTPS

- 機密情報や認証情報を扱う API は HTTPS を前提にする
- 公開 API でも原則 HTTPS を使う

## ブラウザ利用時の考慮

- CORS を適切に設定する
- JSONP は特別な理由がない限り採用しない
- XSS / CSRF / JSON ハイジャック系の脅威を考慮する

## セキュリティヘッダ

必要に応じて次を検討する。

- `Strict-Transport-Security`
- `X-Content-Type-Options: nosniff`
- `Content-Security-Policy`
- `X-Frame-Options`
- `Set-Cookie` の `Secure`, `HttpOnly`, `SameSite`

## レートリミット

- API にはレートリミットを設ける
- ユーザー、IP、API キーなど識別単位を決める
- 制限値、時間窓、超過時レスポンスを明示する

## キャッシュ

- キャッシュ可能な取得系は HTTP キャッシュを検討する
- `Cache-Control`, `ETag`, `If-None-Match`, `Last-Modified` を活用する
- 更新頻度と整合性要求に応じて設定する
