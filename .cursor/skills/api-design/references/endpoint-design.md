# エンドポイント設計

## URI の基本

- 短く、意味が分かり、入力しやすくする
- 人間が読んで理解できる単語を使う
- 省略形を避ける
- 小文字で統一する
- 内部実装や DB 構造を URI に出さない

## リソース命名

- 集合は **複数形の名詞** を使う
- 例: `/users`, `/tasks`
- 個別リソースは `/resources/{id}` で表す
- 名詞で表現できるものは、URI に動詞を入れない

## HTTP メソッド

- `GET`: 取得
- `POST`: 作成
- `PUT`: 全体置換
- `PATCH`: 部分更新
- `DELETE`: 削除

エラー時に `200` を返してボディだけで失敗を表現しない。

## 業務操作

完了・延期・承認のような業務操作は、無理に CRUD に押し込まなくてよい。

- まずはリソース更新として表現できるか検討する
- できない場合は action endpoint を許容する
- action endpoint の具体形はプロジェクト規約で統一する

## クエリパラメータ

- 検索や絞り込みはクエリパラメータを使う
- ページネーションは `limit` / `offset` または `page` / `perPage` のように統一する
- データ更新の意味をクエリパラメータで表現しない

## URI の拡張しやすさ

- Hackable な URI を意識する
- 例: `/products/{id}/reviews`
- 後から機能を足しても崩れにくい構造を選ぶ

## ホスト名と共通パス

- ホストは `api.example.com` 型を優先する
- パス先頭に **メジャーバージョン** を置くのを基本とする
- 例: `/v1/users`

## このリポジトリで確認すべきこと

- リソース指向で表現できるか
- action endpoint が必要か
- 対象の利用者・画面・操作に対して十分に最適化されているか
- 検索・一覧取得・ネストの深さが過剰でないか
