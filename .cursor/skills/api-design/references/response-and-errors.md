# レスポンスとエラー

## データ形式

- 既定は `application/json`
- 複数形式を扱う明確な理由がない限り JSON を基本にする
- `Content-Type` を正しく返す

## レスポンス設計

- クライアントの使いやすさを優先する
- DB テーブル構造をそのまま返さない
- 追加 API 呼び出しを強いる chatty な設計を避ける
- できる限りフラットで一貫した構造にする

## フィールド命名

- 分かりやすい一般語を使う
- 省略形を避ける
- 連結方法は API 全体で統一する
- JSON フィールドの単数 / 複数をデータの形と一致させる
- 短いが意味が分かる名前を選ぶ

## エンベロープ

- 不要なエンベロープは避ける
- HTTP ヘッダで表せる情報は、まず HTTP で表す
- ただし組織の既存規約があるならそれに従う

## ステータスコード

- 成功系は `200`, `201`, `202`, `204` を正しく使い分ける
- クライアントエラーは `4xx`
- サーバエラーは `5xx`
- エラーなのに `200` を返さない

## エラー形式

エラー時は、ステータスコードに加えて機械可読な JSON を返す。

最低限入れる候補:

- `code`
- `message`
- `details`
- `info` または `docsUrl`

## エラー設計の方針

- エラー形式は API 全体で統一する
- ユーザー向け文言と開発者向け情報を混同しない
- HTML エラー画面を返さない

## HTTP 仕様の活用

- `Accept` / `Content-Type` を使う
- キャッシュ可能な取得系では `Cache-Control`, `ETag`, `Last-Modified` を検討する
- ボディ内の独自メタデータで HTTP を置き換えない
