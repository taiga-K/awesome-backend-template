# 原則と利用者別設計

## 中核原則

- 仕様があるものは仕様に従う
- 仕様がないものはデファクトスタンダードに従う
- API は DB や内部アーキテクチャをそのまま露出しない
- API は利用者のユースケースを起点に設計する
- 公開後の変更コストを下げるため、最初から変更しやすさを意識する

## 良い API の判断基準

- 何をする API かが見て分かる
- 覚えやすい
- 一貫性がある
- 変更に強い
- セキュアで堅牢

## このリポジトリの前提

このリポジトリでは、テンプレートとして再利用しやすい API 設計を目指す。  
したがって、特定画面向けの使いやすさと、将来の拡張・複数クライアント対応のバランスを取る。

## 利用者別の考え方

### 既知クライアント中心の API

- 対象画面や操作に最適化したレスポンスを許容する
- 必要なら 1 画面 1 API コールに寄せてよい
- 必要ならオーケストレーション層を設ける

### 公開範囲が広い API

- 予測可能性と一貫性を優先する
- 特定画面専用の都合を外部契約に持ち込みすぎない
- 将来の利用者が初見でも理解できる表現を優先する

## オーケストレーション層を検討する場面

- クライアントごとに必要データが大きく違う
- 1 画面を出すために複数 API を何度も叩く必要がある
- 複数クライアントの最適化要求が強い

## 避けること

- すべてを 1 つの万能 API で解決しようとする
- 厳密な REST という言葉に引きずられて実用性を落とす
- クライアント都合を無視した chatty な API にする
