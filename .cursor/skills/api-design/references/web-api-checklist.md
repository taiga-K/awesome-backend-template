# Web API チェックリスト

設計後は、必ずこのチェックリストで確認する。  
すべてを機械的に満たすことが目的ではなく、**意図して外す項目があるなら理由を説明できること**を重視する。

## URI / エンドポイント

- [ ] URI が短く入力しやすい
- [ ] URI が人間が読んで理解できる
- [ ] URI が小文字のみで構成されている
- [ ] URI が改造しやすい
- [ ] URI にサーバ側のアーキテクチャが反映されていない
- [ ] URI のルールは統一されている
- [ ] 適切な HTTP メソッドを利用している
- [ ] URI で使う単語は、多くの API で同じ意味で使われる一般的な語を選んでいる
- [ ] URI の名詞の単数形 / 複数形が適切
- [ ] URI にスペースや不要なエンコード文字が入っていない
- [ ] URI 内の単語の連結方法が統一されている
- [ ] ページネーションが適切に設計されている

## 認証 / フォーマット

- [ ] ログインや認可の方式が適切に設計されている
- [ ] OAuth 2.0 を使うべき場面では利用を検討している
- [ ] レスポンスのデータ形式は JSON を基本にしている
- [ ] データ形式の指定方法が一貫している
- [ ] 不要な JSONP に対応していない
- [ ] レスポンス内容をクライアントから指定できる必要がある場合、その方法が設計されている

## レスポンスデータ

- [ ] レスポンスデータに不要なエンベロープが入っていない
- [ ] レスポンスデータの構造は可能な限りフラット
- [ ] トップレベルのレスポンス構造が一貫している
- [ ] レスポンスのデータ名として一般的な単語を選んでいる
- [ ] レスポンスのデータ名はなるべく少ない単語数で表現している
- [ ] 複数単語の連結方法が API 全体で統一されている
- [ ] 変な省略形を使っていない
- [ ] 単数形 / 複数形がデータの内容と合っている

## エラー / HTTP

- [ ] エラー時のレスポンスに、原因切り分けに必要な情報が含まれている
- [ ] エラー時に HTML が返らない
- [ ] 適切なステータスコードが返る
- [ ] メンテナンス時に `503` を返す方針になっている
- [ ] 適切なメディアタイプを返している
- [ ] 必要な場合は CORS に対応している

## キャッシュ

- [ ] キャッシュ可能な取得系では `Cache-Control`, `ETag`, `Last-Modified`, `Vary` を検討している
- [ ] キャッシュさせたくないデータに `Cache-Control: no-cache` 等の方針がある

## バージョニング / 提供終了

- [ ] API はバージョンで管理されている
- [ ] API のバージョン方針が明文化されている
- [ ] メジャーバージョン番号が URI に入っており、一目で分かる
- [ ] API の提供終了時の扱いを考慮している
- [ ] API の最低提供期間やサポート方針をドキュメント化している

## セキュリティ

- [ ] HTTPS で API を提供する
- [ ] JSON のエスケープを適切に行う
- [ ] JSON ハイジャック対策が必要な場面を考慮している
- [ ] ブラウザからアクセスする API では XSRF 対策を検討している
- [ ] 受け取るパラメータを不正値まで含めて検証している
- [ ] リクエスト再送時に意図せず再更新しないように設計している
- [ ] セキュリティ対策用の HTTP ヘッダを適切に付与している

## レートリミット

- [ ] レートリミットによる制限を行っている
- [ ] レートリミットの制限回数が想定ユースケースに対して少なすぎない

## 使い方

設計レビュー時は、次の形式で結果をまとめる。

```markdown
チェック結果:
- OK: URI 命名規約、HTTP メソッド、レスポンス形式
- 要確認: action endpoint の表現、キャッシュ方針
- 未対応: レートリミット、提供終了ポリシー

補足:
- 対象利用者とクライアントの前提を明記した
- 一部の項目は現段階では未確定
```
