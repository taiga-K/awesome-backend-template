# 例

## 例1: エンドポイント設計レビュー

**入力**

- ユーザー: `この OpenAPI の path 設計をレビューして`

**期待する動き**

- リソース名が名詞・複数形か確認する
- URI が短く分かりやすいか確認する
- HTTP メソッドが適切か確認する
- action endpoint が必要以上に増えていないか確認する
- 最後に `references/web-api-checklist.md` で抜け漏れを確認する

## 例2: 既知クライアント向け API の判断

**入力**

- ユーザー: `モバイルアプリのホーム画面用 API を設計したい`

**期待する動き**

- 対象クライアントと画面要件を整理する
- 必要なら 1 画面 1 API コールを許容する
- 必要ならオーケストレーション層を提案する

## 例3: レスポンスとエラーの統一

**入力**

- ユーザー: `エラー形式を揃えたい`

**期待する動き**

- ステータスコードを HTTP に沿って整理する
- JSON エラー形式を統一する
- `200` でエラーを返していないか確認する

## 例4: バージョニング判断

**入力**

- ユーザー: `この変更で v2 に上げるべき？`

**期待する動き**

- 後方互換性の有無を確認する
- フィールド追加で済むか確認する
- 破壊的変更ならメジャーバージョンアップを検討する
- 最後にチェックリストで提供終了やサポート方針の観点を確認する
