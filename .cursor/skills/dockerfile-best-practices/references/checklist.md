# Dockerfile Checklist

## 作成前

- `docker/` 配下のどこに置くかを決めたか。迷うなら `docker/Dockerfile`
- コンテナの責務を 1 つに絞れているか
- ビルドコンテキストに不要ファイルがないか。必要なら `.dockerignore` を追加するか

## ベースイメージ

- 公式イメージを優先しているか
- ビルド用と実行用でステージを分けられるか
- 最終イメージを小さく保てるベースを選んでいるか

## レイヤーとキャッシュ

- 変更頻度の低い命令から高い命令へ並んでいるか
- 依存定義ファイルを先に `COPY` しているか
- アプリ本体の `COPY` を後ろに寄せているか
- ビルドに不要な成果物を最終ステージへ持ち込んでいないか

## `RUN`

- 長いコマンドは複数行で読みやすく書いているか
- Debian/Ubuntu 系なら `apt-get update` と `apt-get install` を同じ `RUN` にまとめているか
- `apt` パッケージ一覧は規則的に並び、重複がないか
- パッケージインストール後の不要キャッシュを同じレイヤで削除しているか
- パイプ処理で失敗を見逃さないか

## `COPY` / `ADD`

- 基本を `COPY` にしているか
- `ADD` を使う理由が明確か
- リモート URL 取得を `ADD` に頼っていないか
- 依存ファイルとソース全体のコピーを分けているか

## 実行方法

- `WORKDIR` は絶対パスか
- `CMD` や `ENTRYPOINT` は exec 形式か
- root が不要なら `USER` を設定しているか
- 外部公開ポートがあるなら `EXPOSE` を定義しているか
- `VOLUME` は本当に Dockerfile に必要か。多くのケースではランタイムやオーケストレーション側の volume 設定で足りる
- ヘルスチェックは Dockerfile に持たせるべきか。運用基盤側で管理するほうが自然でないか

## レビュー時の赤信号

- `COPY . .` が先頭付近にあり、毎回キャッシュが壊れる
- ビルド用ツールやソース一式が最終イメージに残っている
- `RUN apt-get update` と `RUN apt-get install` が分かれている
- shell 形式の `CMD` / `ENTRYPOINT` でシグナル処理が不安定
- root 実行のまま本番運用を想定している
- `.dockerignore` がなく、大きなビルドコンテキストを送っている
