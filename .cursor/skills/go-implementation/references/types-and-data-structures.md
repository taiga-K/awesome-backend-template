# 型とデータ構造

## いつ開くか

- ゼロ値で使える API にすべきか迷うとき
- pointer/value receiver や `var` / `:=` の判断に迷うとき
- `slice` / `map` / enum 的表現の落とし穴を確認したいとき

## 基本原則

- ゼロ値が有用な型を優先する
- 型の都合ではなく、意味と誤用しにくさで設計する
- 参照共有やコピーの挙動を明示的に意識する

## 1. ゼロ値の活用

- `sync.Mutex` や `bytes.Buffer` のように、ゼロ値で使える設計を好む
- フィールドがゼロ値で有用なら、親 `struct` もゼロ値で使いやすくなる
- ただし外部リソースを持つ型や unsafe なゼロ値は `New` を検討する

## 2. struct

- リテラルはフィールド名付きで書く
- ゼロ値と同じフィールドは省略し、本当に設定したい値を目立たせる
- `&T{}` の方が `new(T)` より意図が見えやすいことが多い

## 3. ポインタか値か

- 変更する、大きい、`sync.Mutex` を含む型はポインタを基本にする
- 小さく不変に近い値なら value receiver も検討できる
- 同じ型で receiver を混在させない
- 読み取りしかしないのにポインタ引数にしない

## 4. var vs :=

- 非ゼロ初期化なら `:=`、ゼロ値を意図するなら `var` を使う
- `var s []T` は nil slice を明示できる
- `:=` のシャドーイングで外側の `err` を隠していないか注意する

## 5. slice / map

- 空判定は `s == nil` より `len(s) == 0` を優先することが多い
- API 上 `null` と `[]` を区別したいか、JSON の挙動も含めて決める
- nil map は読み取りはできても書き込みで panic する
- `m[k] == 0` だけではキー未存在とゼロ値を区別できないので `comma, ok` を使う
- 部分 slice と `append` は背後配列共有を壊しうる

## 6. enum 的表現

- 独自型と定数群で表す
- 連番が自然なら `iota` を使う
- ゼロ値と「未設定」を区別したいなら、`iota + 1` なども検討する
- 表示やログ用途には `String()` 実装や `stringer` を検討する

## 7. ジェネリクス

- 型だけが違う繰り返しに限って使う
- まずは通常の関数や interface で足りるか確認する
- 抽象化のための抽象化にしない

## レビュー観点

- ゼロ値で安全に使えないのに `New` がないか
- 値/ポインタ receiver が混在していないか
- `sync.Mutex` を含む型を値コピーしていないか
- nil と empty の意味差を API で曖昧にしていないか
- `slice` 共有や `map` 書き込みで意図しない副作用が出ないか
