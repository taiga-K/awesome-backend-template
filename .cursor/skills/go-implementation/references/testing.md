# テスト

## いつ開くか

- table-driven test にすべきか迷うとき
- モックを作り過ぎている気がするとき
- 失敗メッセージや Example の書き方を見直したいとき

## 基本原則

- テストは仕様を固定し、将来の変更を安全にするために書く
- まず標準 `testing` と標準ライブラリで足りるか考える
- 実装詳細ではなく、契約と振る舞いをテストする

## 1. テーブルドリブンテスト

- 入力と期待値の組を並べられるなら、まず table-driven test を検討する
- `t.Run` でケース名を付けると失敗が追いやすい
- テーブル内の分岐が増えたら、別の `TestXxx` に分ける

## 2. 命名と配置

- `_test.go` は対象コードと同じディレクトリに置く
- テスト名は仕様が読める形にする
- フィクスチャやゴールデンは `testdata/` を使う
- `package foo` と `package foo_test` は、内部実装を見たいか公開 API だけ見るかで使い分ける

## 3. 失敗メッセージ

- 入力、got、want を含める
- 何の関数や条件が失敗したかがログだけで分かるようにする
- 文字列一致に寄り過ぎると change detector になりやすい
- エラー判定は `errors.Is` / `errors.As` を優先する

## 4. モックと境界

- モックのためだけに実装側へ大きな interface を置かない
- まず本物に近いテストを検討する
- `httptest` や一時ディレクトリなど、標準ライブラリで作れる実環境を活かす
- テストダブルは振る舞い名で書くと意図が読める

## 5. 継続的に改善する

- バグ修正時は再発防止テストを追加する
- カバレッジは不足箇所発見の補助であり、品質保証そのものではない
- 古いテストや brittle なテストを放置しない

## 6. Example

- `Example` は公開 API のドキュメント手段として有効
- 命名や `Output:` の詳細は [comments.md](comments.md) を主に参照する
- テスト側では、公開 API の使用例として置く価値があるかを判断する

## 7. ツールの扱い

- まず標準 `testing` と手書き stub で足りるか確認する
- `moq` や `gomock` は必要になってから導入する
- フレームワーク都合で interface を大きくしない

## レビュー観点

- コピペしたテストを table-driven にできないか
- ケース名や失敗メッセージが読めるか
- 実装詳細に引きずられた brittle なテストになっていないか
- バグ修正に回帰テストが付いているか
- 本物で確認できるのにモックだけで済ませていないか
