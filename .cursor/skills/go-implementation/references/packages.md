# パッケージ

## いつ開くか

- パッケージの切り方、命名、依存方向に迷うとき
- `internal` / `cmd` / `util` の置き方を見直したいとき
- 循環依存や `init()`、グローバル状態を減らしたいとき

## 基本原則

- パッケージは凝集と結合の単位である
- 名前は短く、役割が読めるものにする
- 依存は最小にし、循環依存を設計のシグナルとして扱う

## 1. 命名と分割

- パッケージ名は単一語を基本にする
- `user.UserService` のような stutter を避ける
- 常に一緒に import されるパッケージは統合を検討する
- 新しいパッケージを増やす前に、既存の凝集を壊していないか確認する

## 2. cmd と internal

- `cmd/<binary>` の `main` は配線と起動に寄せる
- `main` に業務ロジックを置かない
- `internal` は外部へ公開しない実装を閉じ込める手段として使う
- ディレクトリは深くし過ぎず、Go のフラットさを活かす

## 3. 循環依存を避ける

- まず依存方向を見直す
- 振る舞い依存なら interface で反転できる
- 共有データ型なら第3のパッケージへ抽出する
- そもそも分ける必要が弱いなら統合も選択肢にする

## 4. グローバル状態と init

- `init()` は見えにくい副作用を増やすので慎重に使う
- 設定読み込みや外部 I/O を `init()` に置かない
- 可変グローバルより、明示的な生成と DI を優先する
- 例外は、決定的で副作用の小さい初期化に限る

## 5. util / common を作らない

- `util` `common` `base` `shared` は責務の逃がし先になりやすい
- 「その関数は何をしているか」に答えられるパッケージ名へ置き直す
- 一箇所でしか使わない helper は、呼び出し側に置いたままでよい

## 6. 依存を最小にする

- 小さな重複を消すためだけに重い依存を入れない
- 推移的依存も自分たちのコストになる
- 標準ライブラリで足りるなら、まずそれを使う

## 7. シャドーイング

- `path` や `url` などの import 名とローカル変数名を衝突させない
- `import .` はテスト以外では避ける
- 必要なら import alias を使うが、まずローカル変数名を見直す

## レビュー観点

- stutter したパッケージ名や型名になっていないか
- `main` や `init()` に副作用や本体ロジックを抱えていないか
- `util` `common` に責務の違うものを詰め込んでいないか
- 循環依存を雑な interface で隠していないか
- 小さな用途のために重い依存を追加していないか
