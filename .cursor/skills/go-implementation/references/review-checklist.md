# Go レビューチェックリスト

## レビューの優先順位

1. バグ、競合、goroutine リーク、API 誤用の誘発
2. エラー処理、制御フロー、依存方向、責務分割
3. 命名、コメント、テスト不足、軽微な可読性改善

## 指摘の書き方

- まず **現象** を書く
- 次に **なぜ問題か** を書く
- 最後に **直し方の方向性** を書く
- 可能なら「Go の慣用に反する」だけで終わらせず、保守性や誤用リスクに結びつける

詳細な根拠や例は [../SKILL.md](../SKILL.md) の追加資料から該当トピックを開く。

## 1. 命名

- 名前は短いが意味が分かるか
- 型情報の重複や他言語風の命名が混ざっていないか
- `URL` `HTTP` `ID`、getter の扱いが Go 慣習通りか

## 2. エラー

- `panic` で済ませていないか
- 下位層で `log + return` をしていないか
- `%v` で元エラーを潰していないか
- 呼び出し側で必要な区別だけを公開しているか
- `errors.Is` / `errors.As` を使うべき箇所で直接比較していないか
- nil interface の罠を踏んでいないか

## 3. 制御フロー

- 正常系が追いやすいか
- 早期 return でネストを浅くできないか
- `range` 変数のコピーや capture でバグが出ないか
- `defer` で close / unlock / rollback を漏らしていないか
- 長い関数で naked return を使っていないか

## 4. 型と API

- ゼロ値で壊れずに使えるか
- pointer を使う理由があるか
- value receiver / pointer receiver が混在し過ぎていないか
- 返り値が interface になっていて利用者の自由を奪っていないか
- constructor が必須か、ゼロ値で足りるか
- cleanup 責務が API から読み取れるか

## 5. interface

- interface が実装側ではなく利用側の都合で置かれているか
- interface が大き過ぎないか
- モックのためだけの interface 乱造になっていないか
- `accept interfaces, return structs` に反していないか

## 6. パッケージ

- パッケージ名は役割を表しているか
- `common` `util` `base` に責務が混ざっていないか
- 循環依存を生む設計になっていないか
- `init()`、package global、隠れた副作用が増えていないか
- 内部向け実装を unnecessary に export していないか

## 7. テスト

- 仕様の境界条件を押さえているか
- table-driven test にできる箇所で重複したテストを書いていないか
- 失敗時の出力が読めるか
- バグ修正なのに再発防止テストがない、という状態ではないか
- Example や doc comment が有効な公開 API に補助が必要ではないか

## 8. コメント

- export シンボルに doc comment があるか
- doc comment がシンボル名で始まっているか
- 実装詳細を利用者向けコメントへ書き過ぎていないか
- concurrency-safe、`Close` 必須、nil 許容など重要契約が欠けていないか

## 9. 並行処理

- goroutine の終了条件があるか
- `context` が伝播しているか
- channel close の責務が明確か
- receive / send が永遠にブロックしないか
- `sync.Mutex` で十分なのに複雑な並行構造を作っていないか

## 最後に確認する

- 指摘が「慣用に反する」だけで終わらず、保守性や誤用リスクにつながっているか
- 重大な問題と軽微な改善を混ぜず、優先順位が見えるか
- 詳細な根拠や代替案が必要なら、対応する reference を参照したか
