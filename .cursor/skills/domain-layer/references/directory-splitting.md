# `internal/domain` の肥大化判断と分割

## いつ使うか

- `internal/domain` 配下の構成をどう切るか相談されたとき
- ドメイン層が肥大化しているかを判断したいとき
- `model/` と `service/` を分けるべきか議論するとき
- 集約単位、サブドメイン単位、`shared` の切り方を整理したいとき

## 基本原則

- 最初の分割軸は **技術要素ではなく業務境界** にする
- `internal/domain/model` と `internal/domain/service` をトップレベルに置くのは **基本形にしない**
- まず `internal/domain/task`, `internal/domain/user` のように **集約・サブドメイン単位** で切る
- `model/`, `service/`, `policy/`, `event/`, `repository/` は **その業務境界の内側が混み始めたときだけ** 追加する
- ファイル数だけでなく、**責務の見通し・変更時の横断量・境界の濁り** で判断する

## 判断フロー

1. まず責務の境界を確認する
   - これは `task` と `user` のどちらの関心か
   - 集約内のルールか、集約横断の調停か
   - ドメインに置くべきか、ユースケース層に上げるべきか
2. 業務境界の中で追いやすいかを見る
   - その変更を追うとき、同じディレクトリ内で完結するか
   - 新しい人が開いたとき、最初に見るべき場所が明確か
3. 次の肥大化シグナルを確認する
   - 役割の異なるものが同列に並びすぎている
   - 変更のたびに多くのファイルを横断する
   - 名前だけでは責務が見えなくなっている
   - 集約外の都合が入り込み、境界が曖昧になっている
4. 分割先を選ぶ
   - まだ小さい: 業務境界ディレクトリの直下に保つ
   - 同種の概念が増えた: その業務境界の内側で `model/`, `service/`, `event/` などへ分ける
   - 集約をまたぐ: ユースケース層へ上げるか、真に共有概念なら `domain/shared` を検討する

## 肥大化シグナル

次のどれかが見えたら、1 段掘ることを検討する。

- **責務の混在**: エンティティ、値オブジェクト、ドメインサービス、イベント、ポリシー、リポジトリ抽象が増え、一覧しただけで役割が判別しづらい
- **横断量の増加**: 1 つの業務変更で関連ファイルを何度も探し回る
- **理解コストの上昇**: `task/` を開いても何から読むべきか分からない
- **同種ファイルの増加**: 同じ種類のファイルが 3〜4 個以上並び始め、名前だけで区別しにくい
- **命名の劣化**: `helper`, `util`, `domain_service2` のような逃げの命名が増える
- **境界の崩れ**: `task` 配下に `user` 都合の判定や参照解決が大量に入り始める

目安として、1 つの業務境界配下のファイルが 7〜10 個を超え、上のシグナルが複数見えるなら分割候補とみなす。  
ただし、**数だけで分けない**。

## 分割の優先順位

### 1. まず業務境界で切る

```text
internal/domain/
├─ task/
├─ user/
└─ shared/
```

- まずはこの形を基本とする
- `shared/` は本当に複数境界で意味を共有する概念だけに限定する

### 2. 次に業務境界の内側で切る

```text
internal/domain/
└─ task/
   ├─ model/
   ├─ service/
   ├─ policy/
   ├─ event/
   └─ repository/
```

- `model/`: エンティティ、値オブジェクト、ファクトリなど、集約の中核概念
- `service/`: 単一エンティティに閉じないドメインルール
- `policy/`: 判定ルールや仕様分岐が増えたとき
- `event/`: ドメインイベントを扱うとき
- `repository/`: リポジトリ抽象を分けたいとき

### 3. 集約横断はユースケース層を先に疑う

- 複数集約をまたぐ手順、参照解決、副作用の調停は `domain` ではなくユースケース層（application層）が責務であることが多い
- 「`task` と `user` の状態を見て処理順を決める」のようなものは、まずユースケース層へ逃がせないか考える
- このリポジトリでは、ユースケース層の実装先は原則 `internal/usecase/` と考える

## `model/` と `service/` を分ける判断

- **分けなくてよい**:
  - その業務境界配下のファイルが少ない
  - エンティティとドメインサービスを同時に読んだ方が理解しやすい
  - 将来増えそうという予想しか根拠がない
- **分けた方がよい**:
  - `service` が複数になり、役割差が見え始めた
  - `model` の種類が増え、概念の中心を見失いやすい
  - レビュー時に「モデル変更」と「ルール変更」を分けて追いたい

## やってはいけないこと

- `internal/domain/model` と `internal/domain/service` を最初からトップレベルに置く
- `shared`, `common`, `util` を安易な退避場所にする
- 集約横断のオーケストレーションを `domain/service` に押し込む
- HTTP、SQL、ORM、外部 API SDK 由来の都合を `domain` に持ち込む
- ファイル数だけを根拠に機械的にディレクトリを増やす

## 回答の出し方

このテーマで回答するときは、次の順で短く整理する。

1. **結論**: まだ分けない / 業務境界で分ける / 業務境界の内側で分ける
2. **理由**: 肥大化シグナルのどれに当てはまるか
3. **推奨構成**: ディレクトリ例を `text` ブロックで示す
4. **注意点**: ユースケース層へ逃がすべき責務や `shared` の扱い

## 具体例

### まだ分けない例

```text
internal/domain/
└─ task/
```

使う条件:

- `task` 配下の概念が少ない
- 変更が少数ファイルに閉じる
- `model` と `service` を分けるとかえって追いにくい

### 業務境界の内側で分ける例

```text
internal/domain/
└─ task/
   ├─ model/
   ├─ service/
   └─ repository/
```

使う条件:

- `task` 配下の責務が増えた
- レビュー時にモデル変更とルール変更を分けて見たい
- 同種ファイルが複数あり、直下に並べると見通しが悪い

### 分け方を間違えている例

```text
internal/domain/
├─ model/
├─ service/
└─ repository/
```

問題点:

- `task` と `user` の境界がぼやける
- 1 つの業務変更で複数ディレクトリを往復しやすい
- 集約の凝集より技術分類が前に出る

### ユースケース層へ上げるべき例

状況:

- `task` の完了条件判定に加えて `user` の状態確認が必要
- さらに通知送信や永続化順序の調停も必要

判断:

- 集約横断の手順と副作用調停なので、まずユースケース層の責務を疑う
- `domain/service` に押し込まず、`internal/usecase/` から各集約と外部依存を調停する
