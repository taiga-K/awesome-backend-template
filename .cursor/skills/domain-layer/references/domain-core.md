# ドメイン層の中核判断

## この資料が扱う範囲

- ドメイン層にロジックを置く目的
- プリミティブ型と値オブジェクトの使い分け
- 永続化済みデータからの再構成
- リポジトリの責務、配置、更新単位
- 外部 API とドメインの境界
- 集約内外の参照、命名、削除

## 基本姿勢

- **重要なビジネスルールはドメイン層に寄せる**
- **DB やフレームワーク都合でドメインモデルを歪めない**
- **何でも値オブジェクトにするのではなく、課題ドリブンで選ぶ**
- **集約境界を先に決め、リポジトリはその境界を表現する**
- **What は内側、How は外側に置く**

## 回答フロー

1. 問いがどの論点かを切り分ける
   - ドメインオブジェクト
   - リポジトリ
   - 集約境界
   - 外部連携
2. その責務がどの層かを確認する
   - ドメインの不変条件か
   - ユースケースの調停か
   - インフラの実装詳細か
3. 次の優先順位で回答する
   - まず **結論**
   - 次に **理由**
   - 必要なら **置き場所**
   - 最後に **例外やトレードオフ**

## よくある結論

- ドメイン層を独立させる目的は、最重要なルールを外側の都合から守って修正しやすくすること
- 影響範囲の大きさは、ドメインロジックの分散ではなく、ユースケース層のテストで把握する
- 値オブジェクト化は目的があるときに行い、理由のない全面導入はしない
- 再構成用メソッドは、型安全性と実装コストの観点からドメイン層に置く現実解を取ってよい
- リポジトリのインターフェイスは、集約の永続化境界を表すためにドメイン層に置く
- リポジトリ更新は集約単位で行い、一部カラムだけを直接更新する設計は避ける
- 外部 API の値がドメイン概念ならリポジトリで扱い、そうでなければユースケース層側ポートを検討する
- ソート条件は抽象で受け、クエリ組み立てはインフラで行う
- キャッシュはドメインリポジトリに混ぜるより、CQRS の参照モデル側で検討する
- 集約外の参照は ID 参照を基本とし、集約内だけインスタンス参照を許す
- エンティティから直接リポジトリ操作を行わせない
- 単一集約の不変条件はドメインへ、複数集約の調停はまずユースケース層（application層）へ置く
- Go では `NewXxx()` と `Reconstruct()` を分け、生成時ルールと復元を混同しない
- `domain` へ ORM、HTTP、JSON、SQL の表現を持ち込まない

## ドメイン層にロジックを寄せる目的

### 結論

重要なビジネスルールはドメイン層に寄せる。

### 理由

- ドメイン層を DB やフレームワークから独立させ、最も重要なルールを妥協なく表現するため
- 技術都合でモデルを歪めると、設計意図と実装が乖離しやすいため
- 影響範囲はユースケース層のテストで把握し、修正可否を判断できるため

### 注意

- 「何でもドメイン層」は誤り
- 手順の調停や副作用の制御はユースケース層に残す

## プリミティブ型と値オブジェクト

### 結論

すべてを値オブジェクトにする必要はない。

### 判断基準

- 不変条件を表したい
- 概念名を持たせたい
- 同じ型でも意味を区別したい
- 振る舞いを一緒に閉じ込めたい

上のどれかが強くあるなら値オブジェクト化を検討する。  
そうでなければプリミティブ型でもよい。

### 実装時の注意

- 値オブジェクトは可能ならイミュータブルにする
- フォーマット検証や正規化は生成時に終わらせる
- ID、金額、メールアドレス、期間、状態は候補になりやすい

## DB の値からの再構成

### 結論

生成用とは別に、再構成用のメソッドやコンストラクタを設けてよい。

### ポイント

- 作成時の初期状態ルールと、永続化済みデータの再構成は目的が違う
- 再構成用は全属性を受け取り、通常の生成ルールをそのまま適用しない設計を取ってよい
- 名前は `reconstruct`, `fromRepository` など、用途が明確なものにする

### Go での実践

- `NewXxx()` は新規生成用
- `ReconstructXxx()` は DB や外部保存データからの復元用
- 使い分けを曖昧にすると、初期値ルールと復元責務が混ざる

## エンティティの状態変更

### 結論

public setter を並べて状態を作らず、業務語彙のメソッドで変更する。

### 理由

- 不変条件を外から壊しにくくなる
- 状態遷移の意図がコード上で読める
- ユースケースへルールが漏れにくい

### 例

- `ChangePrice()`
- `Postpone()`
- `Complete()`

## 再構成メソッドを書くレイヤー

### 結論

再構成用メソッドをドメイン層に置く現実解は妥当。

### 理由

- インフラ層だけで再構成しようとすると、言語によっては反射や危険な初期化手段に寄りやすい
- ドメイン型自身に再構成口を持たせた方が型安全性を保ちやすい

### 注意

- これはドメイン知識そのものではなく、実装上の現実解
- 言語機能と課題に応じて判断する

## ドメインオブジェクトと DB の対応

### 結論

ドメインオブジェクトとテーブルカラムは 1 対 1 である必要はない。

### ポイント

- ドメイン層はインフラ都合に引きずられない抽象モデルを定義する
- インフラ層は性能、追跡要件、保存戦略に応じて柔軟にマッピングしてよい
- ドメインに無い `updatedAt` のようなカラムを持ってもよい

## ソートの実装方法

### 結論

ソート条件はリポジトリの抽象引数で受け、実際のクエリ組み立てはインフラ実装で行う。

### ポイント

- ドメインやアプリケーションは永続化技術に依存しない表現で条件を渡す
- RDB なら `ORDER BY` 生成はインフラ実装で吸収する

## キャッシュの実装方法

### 結論

ドメインリポジトリへ直接キャッシュを混ぜる前に、CQRS の参照モデルを検討する。

### 理由

- キャッシュは参照性能の問題であり、更新整合性を担う集約リポジトリとは関心が違う
- クエリ用途のモデル側で扱った方が責務が明確になる

## 外部 API とリポジトリの関係

### 結論

外部 API がドメイン概念を取得・保存する窓口ならリポジトリで扱ってよい。  
そうでなければユースケース層側ポートや Adapter を検討する。

### 判断基準

- 取得・送信する値がドメインモデルとして意味を持つか
- それが集約の取得・保存という抽象に乗るか

## 外部 API の値の詰め替え

### 結論

外部 API 呼び出しとドメインオブジェクトへの変換は、インフラのリポジトリ実装内で隠蔽する。

### 理由

- 内側が知りたいのは「何を指定すると何が取れるか」という What だけ
- 呼び出し方法や変換方法という How はインフラの責務

## `domain` に持ち込まないもの

### 結論

`domain` は ORM、HTTP、JSON、SQL の詳細に依存させない。

### 例

- `gorm` tag
- `json` tag
- SQL row struct
- HTTP request / response 用 DTO

## リポジトリのインターフェイスを定義するレイヤー

### 結論

リポジトリのインターフェイスはドメイン層に置く。

### 理由

- リポジトリは集約の永続化単位を表すため
- ユースケースごとに境界が変わると集約設計がぶれるため
- ドメインサービスから参照したい場合にも自然だから

## 一部カラムだけ更新したい場合

### 結論

リポジトリ更新は集約単位で行う。

### 理由

- 集約は強整合性を保つ単位であり、部分更新を直接露出させると整合性の責務が壊れやすい
- 変更は集約ルートへ反映し、その結果を丸ごと保存する

## エンティティからリポジトリを操作してよいか

### 結論

非推奨。

### 理由

- エンティティに永続化責務まで持たせると凝集が下がる
- テストや依存方向が複雑になる

## エンティティ同士の紐付け

### 結論

集約内はインスタンス参照、集約外は ID 参照を基本にする。

### 例

- `Task` と `User` が別集約なら、`Task` が `UserId` を持つ
- 同一集約なら、集約ルートが内部オブジェクトの参照を管理する

### 注意

- 2 つの集約ルートを毎回まとめて更新したくなるなら、集約境界の設計を見直す

## ドメインサービスの命名

### 結論

`Service` を付けなくてもよい。  
ただし、命名規約はチーム内で統一する。

### 例

- `DistanceCalculationDomainService` より `DistanceCalculator` の方が責務が明確なら後者でよい

## ドメインサービスを使う境界

### 結論

エンティティや値オブジェクトへ載せると不自然なルールだけをドメインサービスへ出す。

### 理由

- ドメインサービスは便利な退避先になりやすく、手順と IO が混ざると実質的にユースケース化しやすい
- 複数 repository、外部 API、通知送信、トランザクション制御が集まると、`What` ではなく `How` を持ち始める
- その状態になると、テストはドメイン知識ではなくモック前提のオーケストレーション確認に寄りやすい

### 判断基準

- 単一エンティティに閉じるなら、まずエンティティへ置く
- 複数集約や集合にまたがるドメイン知識なら候補
- 永続化順序や副作用調停が主題ならユースケース層を優先する

### ユースケース層を優先する具体条件

次のどれかに当てはまるなら、まずユースケース層へ置けないかを検討する。

- トランザクション開始、commit、rollback を扱う
- 2 つ以上の repository の呼び順を制御する
- 外部 API、メッセージ送信、通知送信を直接起動する
- retry、timeout、lock、idempotency を扱う
- DTO の組み立てや返却形の調整を行う

### repository 利用を許容する場合の目安

- 集合に対する存在確認や重複確認
- 複数候補の中からドメイン上の成立可否を判定するための参照
- エンティティや値オブジェクトに閉じ込めると不自然な問い合わせ

ただし、repository を使うこと自体を目的化しない。  
domain service が **IO の入口** になりすぎていないかを常に確認する。

## リポジトリ経由の削除

### 結論

削除は ID 指定でよい。

### 理由

- 集合から識別子で要素を削除するという意味で自然
- 削除のためだけに取得コストを払わずに済む

## 回答時の注意

- 「常にこうすべき」と言い切らず、言語仕様や課題の種類による現実解を示す
- ドメイン知識とインフラ都合を混同しない
- 集約の強整合性を崩す提案をしない
- 命名規約は絶対化せず、チーム内で統一することを重視する
- Go の実装相談では、必要なら `references/go-implementation.md` を併読して最小コード例を出す

## 回答テンプレート

1. **結論**: どの層に置くか、何を選ぶか
2. **理由**: なぜその責務なのか
3. **設計ポイント**: 集約、抽象、依存方向のどれを守る話か
4. **例外**: 言語機能や性能要件による現実解
