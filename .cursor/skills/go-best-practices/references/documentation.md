## ドキュメントコメント

全てのトップレベルの公開名にはドキュメントコメントを付ける。重要な非公開型や関数にも付ける。

### 基本ルール

宣言の直前に、空行を挟まずにコメントを配置する。対象の名前で始める:

```go
// Request はコマンド実行リクエストを表す。
type Request struct { ...

// Encode は req の JSON エンコーディングを w に書き込む。
func Encode(w io.Writer, req *Request) { ...
```

### コメントスタイル

Goは `/* */` ブロックコメントと `//` ラインコメントを提供する。ラインコメントが標準。ブロックコメントは主にパッケージコメントや式内、大規模なコードの無効化に使用。

## パッケージコメント

パッケージコメントは、package句の直前に空行なしで配置する:

```go
// Package math は基本的な定数と数学関数を提供する。
package math
```

```go
/*
Package template はテキスト出力（HTMLなど）を
生成するためのデータ駆動テンプレートを実装する。
....
*/
package template
```

### package main のコメント

バイナリ名の後に他のスタイルも許容:

```go
// Binary seedgen ...
package main

// Command seedgen ...
package main

// Seedgen ...
package main
```

小文字で始めるのは許容されない — パッケージコメントは公開表示され、正しい英語で書くべき。

### 長いパッケージコメント

非常に長い場合は `doc.go` を使用する。

## コメント文の書き方

宣言をドキュメントするコメントは完全な文であるべき。対象の名前で始め、ピリオドで終える:

```go
// Compile はregexpを解析し、成功すれば正規表現オブジェクトを返す。
func Compile(str string) (*Regexp, error) {

// Request はコマンド実行リクエストを表す。
type Request struct {
```

### ルール
- godocに抽出された際に適切にフォーマットされるよう完全な文を書く
- これは少し冗長に見えても必要
- フラグメント（文にならない断片）は大文字/句読点不要
- ドキュメントコメントは常に完全な文を要求

## サンプルコード

パッケージを追加する際は、使用例を含める:

### 実行可能なExample関数

テストファイル内に配置し、godocに表示される:

```go
func ExampleFoo() {
    result := Foo("hello")
    fmt.Println(result)
    // Output: HELLO
}
```

### 不可能な場合

コメント内にフォーマット済みのコードスニペットを含める。

## godoc規約

### フォーマット
- 空行で段落を分ける
- 2スペースでインデントした行はそのまま表示（verbatim）
- 大文字で始まる単独の行（後に段落が続く）はヘッダーとしてフォーマット

### プレビュー

ドキュメントはコードレビュー前/中にプレビューする。

### %q フォーマット

引用符が必要な文字列には `%q` を使用。空文字列や制御文字を含む文字列に特に有用:

```go
// 良い例 — 空文字列が明確
t.Errorf("Foo(%q) = %q; want %q", input, got, want)
// 出力: Foo("") = ""; want "hello"

// 悪い例 — 空文字列が不明確
t.Errorf("Foo(%s) = %s; want %s", input, got, want)
// 出力: Foo() = ; want hello
```

## Context関連のドキュメント

Context のキャンセル動作は暗黙的。以下の場合のみドキュメントする:

- キャンセル時に `ctx.Err()` とは異なるエラーを返す場合
- 他のメカニズムが実行を中断する場合
- コンテキストの生存期間や付属値に特別な期待がある場合

---

## 並行性のドキュメント

概念的に読み取り専用の操作は並行使用に安全と仮定される。追加の注記は不要。

変更操作は以下の場合にドキュメントが必要:
- 操作の種類（読み取り vs 変更）が不明確な場合
- APIが同期を提供する場合
- APIがユーザー実装のインターフェースを消費し、並行性要件がある場合

## シグナルブースティング

通常と異なる条件文にはコメントで注意を引く:

```go
if err == nil { // エラーなしの場合
    // 正常処理
}
```

読者が見落としやすい意図的な逸脱を強調するためにコメントを使用する。
