# ユースケース層の責務

## いつ使うか

- ユースケース層に何を置くべきか迷ったとき
- domain と infra の間で何を仲介するか整理したいとき
- DTO、認可、トランザクション、戻り値の境界を決めたいとき

## 基本方針

- ユースケース層は「業務知識の本体」ではなく「アプリケーションの流れの調停役」
- 単一集約の不変条件は domain
- 複数集約、外部依存、認可、入出力変換、トランザクションは usecase
- 複数集約はまず usecase で調停し、それでも集約横断のドメイン知識が残るならドメインサービスやイベントを検討する

## ユースケース層に置くもの

- Input DTO
- Output DTO
- 認可の呼び分け
- トランザクション開始
- 複数 repository の呼び分け
- 外部 API や通知の順序制御
- domain object を使った最終的な戻り値整形

## ユースケース層に置かないもの

- 単一集約の生成ルール
- 値オブジェクトの検証本体
- エンティティ内部の状態遷移ルール
- ORM や SQL の詳細
- HTTP request/response の詳細形式

## まず確認する

1. これは単一集約の不変条件か
2. repository や外部依存を複数またぐか
3. 認可や副作用順序があるか
4. domain object をそのまま返すべきか、DTO に閉じるべきか

## DTO の扱い

- Input DTO は外側の request をそのまま写さず、ユースケースの入力として必要な情報だけに絞る
- Output DTO は外側へ返す境界のために使う
- domain object を外へそのまま返すと、後で presentation 都合が逆流しやすい

## 戻り値の判断

- 内側の別ユースケースやユースケース層内 helper だけが消費するなら domain object でもよい
- presentation 層へ返すなら Output DTO を第一候補にする
- 非同期ジョブ投入の結果など、処理結果を簡潔に返したいなら専用 result struct を使う

## 認可

- 認可はユースケースの入口で確認する
- エンティティの内部ルールと、誰が実行してよいかは別問題として扱う

## ユースケースからユースケースを呼ぶか

- 原則は避ける
- 共通化したいのが「業務ルール」なら domain へ寄せる
- 共通化したいのが「手順」なら private helper へ逃がす前に、処理単位を見直す

## 複数集約の扱い

- 最初の選択肢はユースケース層での調停
- 集約ごとの不変条件は各 domain に残す
- ユースケース層に毎回同じ集約横断ルールが残り続けるなら、責務名ベースのドメインサービスを検討する
- 副作用連携や後続処理の切り出しが中心なら、ドメインイベントや非同期処理も候補にする

## 回答の出し方

1. 結論
2. domain と usecase の境界
3. 置き場所
4. DTO/戻り値の形
5. 注意点
