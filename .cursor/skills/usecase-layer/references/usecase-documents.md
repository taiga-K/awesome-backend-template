# `docs/domain/usecase` の扱い

## いつ使うか

- ユースケース文書を作るとき
- 実装変更に合わせて `docs/domain/usecase/` を更新するとき
- ドメインモデル図や API 設計との同期範囲を確認したいとき

## 基本方針

- ユースケース文書は「誰が」「何をしたいか」「どこまで完了か」を記述する
- 実装詳細より、アクター、入力、主成功シナリオ、代替シナリオ、事後条件を優先する
- ドメイン知識の本体はドメイン文書へ、HTTP 詳細は API 文書へ分ける

## 最低限そろえるもの

- アクター
- トリガー
- 前提条件
- 主成功シナリオ
- 失敗・代替パス
- 事後条件
- 関連するドメイン概念

## 同期が必要なもの

- ドメイン概念が変わるなら `docs/domain/uml/`
- 用語が変わるなら `docs/domain/ubiquitous-language.md`
- 外部公開 API が変わるなら `docs/api/`

## 避けたいこと

- 実装コードをそのまま文書へ貼る
- HTTP path や status code をユースケース文書の中心にする
- ドメインの制約と UI の制約を混ぜる

## 回答の出し方

1. 更新すべきユースケース文書
2. 同期が必要な周辺文書
3. まだ未確定の点
