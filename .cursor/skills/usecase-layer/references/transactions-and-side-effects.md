# トランザクションと副作用

## いつ使うか

- DB 更新と外部 API 呼び出しが同じユースケースにあるとき
- 通知、メール、ジョブ投入をどの順序で実行すべきか決めたいとき
- 失敗時のロールバックや再実行方針を整理したいとき

## 基本方針

- トランザクション境界はユースケース層に置くのを第一候補にする
- DB の強整合と、外部副作用の最終整合は分けて考える
- 「どこまで同期で完了させるか」を先に決める
- 副作用は順序と失敗時の扱いを明示する

## まず確認する

1. 同期で成功させるべき範囲はどこまでか
2. 外部 API 失敗時に DB 変更を戻すべきか
3. リトライで二重実行されても安全か
4. 通知やイベント投入は非同期へ逃がせるか

## 推奨順序の考え方

### 典型パターン

1. 認可
2. transaction 開始
3. domain object 生成・更新
4. repository 保存
5. transaction 完了
6. 非同期イベント投入 or 通知

理由:

- DB の整合性を先に確定する
- 外部副作用の失敗で主処理まで巻き添えにしにくい

## 同期実行にしてよい場面

- 外部 API 成功がビジネス上の完了条件そのもの
- その API が失敗したら処理全体を失敗扱いにすべき
- 再実行や二重送信に対する保護がある

## 非同期を優先したい場面

- 通知、メール、監査ログ、検索インデックス更新
- 主業務の完了とは独立してよい副作用
- 一時失敗しても後追い回復できる処理

## 避けたいこと

- transaction の中で長い外部 API 呼び出しを待つ
- 副作用の失敗方針を決めずに実装する
- 「失敗したらもう一度押せばいい」で済ませる
- 同じユースケースで DB 保存、外部 API、メール、監査ログを全部同期でつなぐ

## 回答の出し方

1. どこまで同期で保証するか
2. transaction 境界
3. 副作用の順序
4. 失敗時の扱い
5. リトライや非同期化の候補
