# Infrastructure as Code（IaC）セキュリティ

- `rule_id`: `codeguard-0-iac-security`
- 原題: `Infrastructure as Code (IaC) Security`

Terraform や CloudFormation などの言語でクラウド基盤を設計し Infrastructure as Code（IaC）を記述するときは、公開露出の防止や最小権限の原則など、常に安全な慣行とデフォルトを用いること。設定ミスを積極的に特定し、安全な代替案を示すこと。

## IaC における重要なセキュリティパターン

### ネットワークセキュリティ
- **常に** リモート管理サービス、データベース、LDAP、TACACS+ など機密性の高いサービスへのアクセスを制限すること。必要がない限り、いかなるサービスもインターネット全体からアクセス可能にしてはならない。代わりに、アクセスが必要な IP アドレスまたは CIDR ブロックの集合に限定すること。
    - セキュリティグループおよび ACL のインバウンドルールで、リモート管理ポート（SSH 22、RDP 3389 など）に対して **`0.0.0.0/0` を許可してはならない**。
    - セキュリティグループおよび ACL のインバウンドルールで、データベースポート（3306、5432、1433、1521、27017 など）に対して **`0.0.0.0/0` を許可してはならない**。
    - Kubernetes API エンドポイントの許可リストに **`0.0.0.0/0` を含めてはならない**。EKS、AKS、GKE およびその他の Kubernetes API エンドポイントは、管理アクセスが必要な CIDR アドレスの許可リストに限定すること。
    - クラウドのマネージドデータベースサービス（RDS、Azure SQL、Cloud SQL など）を、すべての IP アドレス `0.0.0.0/0` に公開してはならない。
- 原則として、パブリックなネットワークアクセスが不要な限り、内部 VPC、VNET、VPN、その他の内部トランジットなどプライベートネットワークを優先すること。
- **常に** ネットワーク監視とセキュリティ分析のために VPC/VNET フローログを有効にすること。
- **常に** デフォルト拒否と、必要なトラフィックのみを明示的に許可するルールを実装すること。
- 原則として、インターネットへのエグレスをデフォルトでブロックすることを優先すること。エグレスが必要な場合は、例えば次のような適切なトラフィック制御を検討する。
    - 必要なサービスへのアクセスのみを許可するルールを持つエグレスファイアウォールまたはプロキシ。
    - 必要な IP または CIDR ブロックへのアクセスのみを許可するルールを持つエグレスセキュリティグループ（SG）またはアクセスコントロールリスト（ACL）。
    - 悪意のあるドメインへのアクセスを防ぐ DNS フィルタリング。

### データ保護
- **常に** データベース、ファイルシステム、オブジェクトストレージ、ブロックストレージを含むすべてのストレージサービスで保存時の暗号化を構成すること。
    - クラウドストレージサービス（S3、Azure Blob、GCS バケット）の暗号化を有効にする。
    - すべてのデータベースエンジン（RDS、Azure SQL、Cloud SQL、DocumentDB など）で保存時の暗号化を構成する。
    - 仮想マシンのストレージボリュームについて EBS／ディスク暗号化を有効にする。
- **常に** すべてのデータ通信で転送時の暗号化を構成すること。
    - すべての HTTPS／API 通信に TLS 1.2 以上を用いる。
    - 証明書検証付きでデータベース接続に SSL/TLS を構成する。
    - VPC／VNET 内のサービス間通信の暗号化を有効にする。
    - リモートアクセスには暗号化されたプロトコル（SSH、HTTPS、安全な RDP）を用いる。
- **常に** 機密レベルに応じたデータ分類と保護コントロールを実装すること。
    - PII、PHI、金融データ、知的財産には、より厳格な暗号化とアクセス制御を適用する。
    - データ分類レベルごとに別々の暗号化キーを用いる。
- **常に** 安全なデータ保持と廃棄の方針を構成すること。
    - 規制要件とビジネス要件に基づき保持期間を定義する。
    - 安全な削除を伴う自動化されたデータライフサイクル管理を実装する。
- **常に** 包括的なデータアクセスの監視と監査を有効にすること。
    - すべてのデータアクセス、変更、削除操作をログに記録する。
    - 異常なデータアクセスパターンやデータ流出の可能性を監視する。
    - 機密データへの違反アクセスに対するリアルタイムアラートを実装する。
- **常に** データバックアップを暗号化すること。
    - 本番データとは別の暗号化キーですべてのバックアップデータを暗号化する。
    - バックアップは適切なアクセス制御のもと、地理的に分散した場所に保管する。
    - バックアップの復元手順を定期的にテストし、バックアップの整合性を検証する。

### アクセス制御
- 公開分類としてラベル付けされているか、意図的に公開する場合を除き、**決して** バックアップ、ストレージ、コンテナレジストリ、ファイル共有などの重要な管理またはデータサービスを匿名アクセスのままにしてはならない。
- IAM ポリシーやクラウド RBAC でワイルドカード権限（`"Action": "*"`、`"Resource": "*"`）を **用いてはならない**。
- 不要な場合にサービスアカウントに Owner／Admin ロールを付与し過ぎては **ならない**。
- サービス API キーやクライアントシークレットを用いず、長期クレデンシャルを不要にするロールベースのアクセス制御を伴うワークロードアイデンティティを **用いること**。
- AWS でレガシーの Instance Metadata Service バージョン 1（IMDSv1）を **有効にしたり使用したりしてはならない**。
- OAuth などより安全な代替がある場合に、ローカルユーザーなどレガシーまたは旧式の認証方式を **用いてはならない**。

### コンテナおよび VM イメージ
- ハードニングされていない VM およびコンテナイメージを **用いてはならない**。
- **常に** distroless または最小限のコンテナイメージを選ぶこと。
- 信頼できるソースからの安全なベースライン仮想マシンイメージの利用を **推奨** する。
- 信頼できるソースからの最小限の distroless コンテナイメージの利用を **推奨** する。

### ロギングおよび管理者アクセス
- 機密サービスに対する管理アクティビティのログを **無効にしてはならない**。
- **常に** 特権操作に対する監査ログを有効にすること。

### シークレット管理
- シークレット、パスワード、API キー、証明書を IaC のソースコードに直接 **ハードコードしてはならない**。
- **常に** Terraform ではシークレットに `sensitive = true` を付け、その他の IaC では機密値を示す適切な注釈またはメタデータを用いること。

### バックアップとデータ復旧
- 保存時および転送時の暗号化なしにバックアップを **作成してはならない**。
- **常に** クロスリージョンレプリケーション付きで、バックアップのマルチリージョン保管を構成すること。
- 保持ポリシーとライフサイクル管理なしにバックアップを **構成してはならない**。
