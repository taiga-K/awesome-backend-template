# セッション管理とクッキー

- `rule_id`: `codeguard-0-session-management-and-cookies`
- 原題: Session management and secure cookies (rotation, fixation, timeouts, theft detection)

固定化、乗っ取り、盗難を防ぎつつ、使いやすさを維持する、堅牢で攻撃に強いセッション処理を実装すること。

### セッション ID の生成と性質
- セッション ID は CSPRNG で生成する。エントロピーは 64 ビット以上（128 ビット以上を推奨）。不透明で推測困難かつ意味を持たないこと。
- クッキー名はフレームワークのデフォルトより汎用名（例: `id`）を使う。サーバーが発行していない ID は拒否する。
- セッションデータはすべてサーバー側に保存する。トークンに PII や権限を埋め込まない。機密性が高い場合は、サーバー側セッションストアを保存時に暗号化する。

### クッキーのセキュリティ設定
- セッションクッキーに `Secure`、`HttpOnly`、`SameSite=Strict`（フロー上必要なら `Lax`）を設定する。
- `Path` と `Domain` でクッキーのスコープを狭くする。サブドメイン横断の露出を避ける。
- 非永続のセッションクッキーを優先する（Expires/Max-Age なし）。HTTPS を全面的に要求し、サイト全体で HSTS を有効にする。

ヘッダの例:
```
Set-Cookie: id=<opaque>; Secure; HttpOnly; SameSite=Strict; Path=/
```

### セッションのライフサイクルとローテーション
- セッションはサーバー側でのみ作成する。提示された ID は信頼できない入力として扱う。
- 認証時、パスワード変更時、権限昇格のたびにセッション ID を再生成する。以前の ID は無効化する。
- フレームワークの都合で必要なら、認証前と認証後で別のクッキー名を使う。

### 期限切れとログアウト
- アイドルタイムアウト: 高価値なら 2〜5 分、リスクが低ければ 15〜30 分。絶対タイムアウト: 4〜8 時間。
- タイムアウトはサーバー側で強制する。サーバー上のセッションを完全に無効化し、クライアント側のクッキーも消去する、目に見えるログアウトボタンを提供する。

### 転送とキャッシュ
- セッションの経路全体で HTTPS を強制する。1 つのセッションで HTTP と HTTPS を混在させない。
- セッション識別子や機密データを含むレスポンスには `Cache-Control: no-store` を送る。

### クッキー盗難の検知と対応
- 確立時にサーバー側でセッション文脈をフィンガープリント化する（IP、User-Agent、Accept-Language、利用可能なら関連する `sec-ch-ua`）。
- 着信リクエストを保存したフィンガープリントと比較し、無害な変化（サブネット変更、UA 更新など）は許容する。
- リスクに応じた対応:
  - 高リスク: 再認証を要求する。セッション ID をローテーションする。
  - 中リスク: ステップアップ検証（チャレンジ）。セッション ID をローテーションする。
  - 低リスク: 不審な活動をログに記録する。
- 乗っ取りの可能性が検知されたときは、常にセッション ID を再生成する。

### クライアント側ストレージ
- XSS リスクのため、セッショントークンを `localStorage` / `sessionStorage` に保存しない。転送には HttpOnly クッキーを優先する。
- セッション以外の秘密をクライアント側ストレージに置くやむを得ない場合は、Web Worker で隔離し、ページコンテキストに露出させない。

### フレームワークと複数クッキーのケース
- 組み込みのセッションフレームワークを優先し、更新と堅牢化を継続する。
- 複数のクッキーがセッション状態に関与する場合は関係を検証する。パス/ドメイン間で同じクッキー名を避ける。

### 監視とテレメトリ
- セッション ID の生値ではなく、ソルト付きハッシュを用いてセッションのライフサイクルイベント（作成、ローテーション、終了）をログに記録する。
- セッション ID のブルートフォースや、異常な同時利用を監視する。

### 実装チェックリスト
1) CSPRNG によるセッション ID（エントロピー 64 ビット以上）、不透明かつサーバー発行のみ。
2) クッキーフラグ: `Secure`、`HttpOnly`、`SameSite` を設定。ドメイン/パスを絞る。
3) HTTPS のみと HSTS。混合コンテンツなし。
4) 認証および権限変更時に ID を再生成。旧 ID を無効化。
5) アイドルおよび絶対タイムアウトをサーバー側で強制。完全なログアウトを実装。
6) 機密レスポンスに `Cache-Control: no-store`。
7) サーバー側フィンガープリントと、異常に対するリスクベースの対応。
8) セッショントークンのクライアント保存なし。フレームワークのデフォルトを堅牢化。
