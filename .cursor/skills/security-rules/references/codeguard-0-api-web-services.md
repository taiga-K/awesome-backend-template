# API と Web サービスのセキュリティ

- `rule_id`: `codeguard-0-api-web-services`
- 原題: `API & Web Services Security`

REST、GraphQL、SOAP/WS サービスを、通信路、認証、認可、スキーマ検証、SSRF 対策、DoS 制限、マイクロサービス向け安全パターンまで含めてエンドツーエンドで保護する。

## 通信路と TLS

- `HTTPS` のみを使用する。高価値または内部向けサービスでは `mTLS` を検討する。証明書 (`CN` / `SAN`、失効) を検証し、mixed content を防ぐ。

## 認証とトークン

- クライアントには標準フロー (`OAuth2` / `OIDC`) を使い、独自方式は避ける。サービス間では `mTLS` または署名付きサービス・トークンを使う。
- `JWT` では、アルゴリズムを固定し、`iss` / `aud` / `exp` / `nbf` を検証し、短寿命にし、ローテーションし、ログアウトや失効時には denylist を適用する。失効が必要で中央ストアを利用できるなら opaque token を優先する。
- API キーは権限を狭く限定し、レート制限と使用監視を行い、機微な操作で単独使用しない。

## 認可

- エンドポイント単位、リソース単位のチェックをサーバー側で強制し、デフォルト拒否にする。
- マイクロサービスでは、ゲートウェイで粗い認可、各サービスで細かい認可を行う。外部トークンを内部にそのまま流さず、署名付き内部 ID を伝搬する。

## 入力とコンテンツ処理

- 契約ベースで入力を検証する。例: `OpenAPI` / `JSON Schema`、`GraphQL SDL`、`XSD`。未知フィールドや過大なペイロードは拒否し、上限を設定する。
- `Content-Type` と `Accept` は明示的に強制し、未対応の組み合わせは拒否する。XML パーサーは `XXE` と entity expansion に対して堅牢化する。

## Resolver と Handler における SQL / Injection 安全性

- パラメータ化クエリまたは ORM の bind parameter を用い、ユーザー入力をクエリやコマンドへ文字列連結しない。

## GraphQL 固有の制御

- クエリ深度と総複雑度を制限し、ページネーションを強制し、実行タイムアウトを設定し、本番では introspection と IDE を無効化する。
- フィールド単位、オブジェクト単位の認可を実装して `IDOR` / `BOLA` を防ぎ、バッチ処理を検証し、オブジェクト種別ごとにレート制限を行う。

## 外向き呼び出しの SSRF 防止

- 生の URL を受け取らない。ライブラリでドメインと IP を検証し、`HTTP` / `HTTPS` のみを許可し、`file://`、`gopher://`、`ftp://` などは遮断する。
- ケース 1 固定パートナー: 厳格な allow-list、リダイレクト無効化、ネットワーク出口側 allow-list を用いる。
- ケース 2 任意宛先: `private` / `link-local` / `localhost` 帯を遮断し、解決されたすべての IP が public であることを確認し、可能なら宛先側の署名付きトークンを要求する。

## SOAP/WS と XML の安全性

- SOAP ペイロードを `XSD` で検証し、メッセージサイズを制限し、必要に応じて XML 署名と暗号化を有効化する。
- パーサーを `XXE`、entity expansion、再帰的ペイロードに耐えるよう設定し、添付ファイルをスキャンする。

## レート制限と DoS

- IP、ユーザー、クライアントごとの制限、circuit breaker、タイムアウトを適用する。サーバー側バッチ処理とキャッシュで負荷を下げる。

## 管理用エンドポイント

- インターネットへ公開しない。強い認証 例: MFA、ネットワーク制限、分離されたポートまたはホストを要求する。

## テストと評価

- 正式な API 定義を維持し、その仕様から契約テストと fuzzing を駆動する。
- 各エンドポイントについて、認証・認可バイパス、SSRF、injection、情報漏えいを評価し、トークン検証失敗をログに残す。

## マイクロサービス実践

- コード化されたポリシーと埋め込み判断点を使う。sidecar またはライブラリ型の PDP を検討する。
- サービス ID には `mTLS` または署名付きトークンを使い、外部トークンを内部で再利用しない。
- 相関 ID 付きの集中構造化ログを使い、機微データをサニタイズする。

## 実装チェックリスト

- `HTTPS/mTLS` を設定し、証明書を管理し、mixed content をなくす。
- エッジとサービスの両方で契約検証を行い、未知フィールドを拒否し、サイズと時間の制限を強制する。
- エンドポイントごとに強い認証と認可を実装し、GraphQL の制限を適用し、本番では introspection を無効化する。
- アプリ層とネットワーク層の両方で SSRF 対策を行い、リダイレクトを無効化し、可能な限り allow-list を使う。
- レート制限、circuit breaker、耐障害パターンを導入する。
- 管理用エンドポイントを分離し、強い認証を要求する。
- ログは構造化し、相関 ID を付与し、プライバシー安全にする。

## テスト計画

- スキーマ準拠の契約テストと、スキーマ対応ツールによる fuzzing を行う。
- SSRF、`IDOR` / `BOLA`、認可バイパスに対するペネトレーションテストと、DoS 制限に対する性能試験を行う。
- 各エンドポイントの全 HTTP メソッドをテストし、URL パス、ヘッダー、構造化データ中のパラメータも洗い出す。
- トークン検証と失効挙動に対する自動検査を行う。
