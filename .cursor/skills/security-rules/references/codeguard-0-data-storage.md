# データベースセキュリティのガイドライン

- `rule_id`: `codeguard-0-data-storage`
- 原題: `Database Security Guidelines`

本ルールは、SQL／NoSQL データベースを安全に構成し、データ漏えいや不正アクセスから保護するための助言である。

- バックエンドのデータベース保護
  - データベースサーバを他システムから分離し、ホスト接続を制限する。
  - 可能ならネットワーク（TCP）アクセスを無効にし、ローカルソケットファイルまたは名前付きパイプを使う。
  - 適切な場合はデータベースを localhost のみにバインドする。
  - ファイアウォールで特定ホストへのポートアクセスのみに制限する。
  - データベースサーバをアプリサーバから分離した別 DMZ に置く。
  - 太いクライアントからバックエンド DB への直接接続を許可しない。

- トランスポート層のセキュリティ
  - 暗号化接続のみを許可するようデータベースを構成する。
  - データベースサーバに信頼できるデジタル証明書をインストールする。
  - クライアント接続に TLSv1.2 以上と現代的な暗号（AES-GCM、ChaCha20）を使う。
  - クライアントアプリでデジタル証明書の有効性を検証する。
  - 初期認証だけでなく、すべての DB トラフィックを暗号化する。

- 安全な認証の構成
  - ローカルサーバ接続を含め、常に認証を必須とする。
  - 強くユニークなパスワードでアカウントを保護する。
  - アプリまたはサービスごとに専用アカウントを使う。
  - 必要最小限の権限のみを構成する。
  - アカウントと権限を定期的に見直す。
  - アプリが廃止されたらアカウントを削除する。
  - 退職や侵害の疑いがある場合はパスワードを変更する。

- データベース認証情報の保管
  - アプリケーションのソースコードに認証情報を置かない。
  - Web ルート外の設定ファイルに保管する。
  - 認証情報アクセスに適切なファイル権限を設定する。
  - 認証情報ファイルをソースリポジトリにコミットしない。
  - 利用可能なら組み込み機能で認証情報ストレージを暗号化する。
  - 環境変数またはシークレット管理ソリューションを使う。

- 安全な権限管理
  - すべての DB アカウントに最小権限の原則を適用する。
  - 組み込みの root、sa、SYS などのアカウントは使わない。
  - アプリアカウントに管理権限を与えない。
  - アカウント接続を許可ホストのみに制限する。
  - 開発、UAT、本番でデータベースとアカウントを分ける。
  - 必要な権限（SELECT、UPDATE、DELETE など）のみ付与する。
  - 権限昇格を防ぐため、アカウントをデータベース所有者にしない。
  - 必要に応じてテーブル／列／行レベルの権限を実装する。

- データベースの構成と堅牢化
  - 必要なセキュリティ更新とパッチを定期的に適用する。
  - 権限の低いユーザアカウントで DB サービスを実行する。
  - デフォルトアカウントとサンプル DB を削除する。
  - トランザクションログをメイン DB ファイルとは別ディスクに置く。
  - 適切な権限で定期的な暗号化バックアップを構成する。
  - 不要なストアドプロシージャと危険な機能を無効化する。
  - DB アクティビティの監視とアラートを実装する。

- プラットフォーム固有の堅牢化
  - SQL Server: `xp_cmdshell`、CLR 実行、SQL Browser サービス、混合モード認証（不要なら）を無効化する。
  - MySQL／MariaDB: `mysql_secure_installation` を実行し、ユーザの FILE 権限を無効化する。
  - PostgreSQL: PostgreSQL のセキュリティ文書のガイドラインに従う。
  - MongoDB: MongoDB のセキュリティチェックリストの要件を実装する。
  - Redis: Redis のセキュリティガイドの推奨に従う。

要約:  
データベースシステムを分離し、暗号化接続を強制し、強固な認証を実装し、シークレット管理で認証情報を安全に保管し、最小権限を適用し、DB 構成を堅牢化し、定期的なセキュリティ更新と監視を維持する。
