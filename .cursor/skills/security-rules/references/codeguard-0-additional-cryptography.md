# 追加の暗号化と TLS

- `rule_id`: `codeguard-0-additional-cryptography`
- 原題: `Additional Cryptography & TLS`

保存データと通信データの両方に対して、現代的で検証済みの暗号を適用する。鍵を安全に管理し、TLS を正しく設定し、HSTS を展開し、適切な場合にのみピンニングを検討する。

## アルゴリズムとモード

- 共通鍵暗号: `AES-GCM` または `ChaCha20-Poly1305` を推奨する。`ECB` は避ける。`CBC` と `CTR` は encrypt-then-MAC と組み合わせる場合のみ使用する。
- 公開鍵暗号: `RSA 2048bit` 以上、または現代的な ECC (`Curve25519` / `Ed25519`) を用いる。RSA 暗号化には `OAEP` を使う。
- ハッシュ: 完全性用途には `SHA-256` 以上を用いる。`MD5` と `SHA-1` は避ける。
- 乱数生成: プラットフォームに適した CSPRNG を使う。例: `SecureRandom`、`crypto.randomBytes`、`secrets` モジュール。非暗号学的 RNG は決して使わない。

## 鍵管理

- 鍵は検証済みモジュール (`HSM` / `KMS`) 内で生成し、パスワードや予測可能な入力から生成しない。
- 鍵は用途ごとに分離する。例: 暗号化、署名、ラップ。漏えい、暗号寿命、またはポリシーに応じてローテーションする。
- 鍵は `KMS` / `HSM` / vault に保管する。ハードコードしない。平文の環境変数も避ける。`KEK` で `DEK` をラップし、別々に保存する。
- 信頼ストアへのアクセスを制御し、更新を検証し、すべての鍵アクセスと鍵操作を監査する。

## 保存データ

- 機微データは暗号化する。保存する秘密情報を最小化し、可能ならトークナイズする。
- 認証付き暗号を使用し、nonce と IV を正しく管理し、salt は項目ごとに一意にする。
- バックアップを保護する。暗号化し、アクセスを制限し、リストアを検証し、保持期間を管理する。

## TLS 設定

- プロトコル: `TLS 1.3` を優先する。レガシー互換性が必要な場合のみ `TLS 1.2` を許可する。`TLS 1.0` / `TLS 1.1` / `SSL` は無効化する。`TLS_FALLBACK_SCSV` を有効化する。
- 暗号スイート: `AEAD` スイートを優先し、`NULL` / `EXPORT` / `anon` を無効化する。ライブラリを最新に保ち、圧縮を無効化する。
- 鍵交換グループ: `x25519` / `secp256r1` を優先し、必要に応じて安全な `FFDHE` グループを設定する。
- 証明書: 2048bit 以上の鍵、`SHA-256`、正しい `CN` / `SAN` を使用する。ライフサイクルと失効 (`OCSP stapling`) を管理する。
- アプリケーション: サイト全体を `HTTPS` 化し、`HTTP -> HTTPS` にリダイレクトし、mixed content を防ぎ、Cookie に `Secure` を設定する。

## HSTS

- `Strict-Transport-Security` は必ず `HTTPS` 上でのみ送信する。段階的に展開する。
- テスト: 短い `max-age` 例: `86400` と `includeSubDomains`
- 本番: `1 年以上` の `max-age`。安全であれば `includeSubDomains` を付与する。
- 任意: 十分に成熟したら preload を検討する。恒久性とサブドメインへの影響を理解したうえで適用する。

## ピンニング

- ブラウザの `HPKP` は避ける。制御可能なクライアント 例: モバイル で、かつ両端を自社管理している場合にのみピンニングを検討する。
- `SPKI` ピンニングを予備ピン付きで優先し、安全な更新経路を計画し、ユーザーによるバイパスを許可しない。
- ローテーションと障害時処理を徹底的に検証し、運用リスクを理解する。

## 実装チェックリスト

- すべてで `AEAD` を使い、検証済みライブラリのみを使い、独自暗号を実装しない。
- 鍵は `KMS` / `HSM` で生成・保管し、用途を分離し、ローテーション手順を文書化する。
- `TLS 1.3/1.2` と強力な暗号を使い、圧縮を無効化し、`OCSP stapling` を有効化する。
- HSTS は段階的な計画に沿って導入し、mixed content を排除する。
- ピンニングは正当な理由がある場合のみ使用し、予備ピンと更新経路を用意する。

## テスト計画

- プロトコル、暗号、HSTS を対象に、自動構成スキャン 例: `SSL Labs`、`testssl.sh` を実施する。
- 暗号 API の誤用に対するコードレビューを行い、鍵ローテーションとバックアップ復旧をテストする。
- ピンニングを導入している場合は、ローテーションと障害時のシミュレーションを実施する。
