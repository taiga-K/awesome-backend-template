# 認証と MFA

- `rule_id`: `codeguard-0-authentication-mfa`
- 原題: `Authentication & MFA`

資格情報攻撃に耐え、秘密情報を保護し、フィッシング耐性の高い強力な MFA と安全なリカバリを支える、回復力があり使いやすい認証システムを構築する。

## アカウント識別子と UX

- 公開されない、ランダムで一意な内部ユーザー識別子を使う。ログインには検証済みメールアドレスまたはユーザー名を許可する。
- 常に汎用的なエラーメッセージ 例: `Invalid username or password` を返す。アカウント列挙を防ぐため、処理時間も一定に保つ。
- パスワードマネージャーをサポートする。`<input type="password">` を使い、貼り付けを許可し、JavaScript による妨害をしない。

## パスワードポリシー

- パスフレーズと完全な Unicode を受け入れ、最小 8 文字とし、複雑な構成ルールは避ける。最大長は合理的な値 64 文字以上 のみにする。
- 新しいパスワードは漏えい済みコーパス 例: k-anonymity API で確認し、漏えい済みまたは一般的なパスワードを拒否する。

## パスワード保存 ハッシュ化

- 暗号化ではなくハッシュ化する。ユーザーごとに一意な salt を用い、遅くてメモリハードなアルゴリズムと定数時間比較を使う。
- 推奨順とパラメータ 使うハードウェアに合わせて調整し、サーバー側で 1 秒未満を目標:
- `Argon2id`: `m=19-46 MiB`、`t=2-1`、`p=1` または同等の安全性トレードオフ
- `scrypt`: `N=2^17`、`r=8`、`p=1` または同等
- `bcrypt` レガシーのみ: cost `10` 以上。72 バイト入力制限に注意する。
- `PBKDF2` FIPS: `PBKDF2-HMAC-SHA-256 >= 600k`、または `SHA-1 >= 1.3M`
- 任意の pepper を使う場合は DB 外部 `KMS` / `HSM` に保管し、`HMAC` または pre-hashing で適用する。pepper をローテーションするならユーザー再設定計画を持つ。
- Unicode と null byte はライブラリ全体で end-to-end にサポートされていなければならない。

## 認証フローの強化

- すべての認証エンドポイントとトークン伝送に `TLS` を強制し、`HSTS` を有効化する。
- IP ごと、アカウントごと、全体のレート制限を実装し、proof-of-work や CAPTCHA は最後の手段としてのみ使う。
- ロックアウトとスロットリングには段階的バックオフを用い、永久ロックアウトを避けるためリセットや通知を提供する。
- オラクルやタイミングの手掛かりを減らすため、応答とコードパスを均一化する。

## 多要素認証 MFA

- 機微なアカウントでは、フィッシング耐性の高い要素を既定にする。`passkeys` / `WebAuthn` (`FIDO2`) またはハードウェア `U2F` を使う。
- 許容できる要素: `TOTP` アプリ型、PIN 付きスマートカード。機微用途では `SMS` / 音声 / メールコードは避ける。秘密の質問には依存しない。
- MFA を要求する場面: ログイン、パスワードやメール変更、MFA 無効化、権限昇格、高価値取引、新しい端末や場所からのアクセス。
- リスクベース MFA のシグナル: 新しい端末、地理的速度異常、IP 評判、異常時刻、漏えい済み資格情報。
- MFA リカバリでは、単回使用バックアップコードを提供し、複数要素の登録を促し、リセット時には強い本人確認を要求する。
- MFA 失敗時は、登録済み代替手段を提示し、失敗をユーザーへ通知し、文脈情報 秘密情報なし をログに残す。

## フェデレーションとプロトコル OAuth 2.0 / OIDC / SAML

- 標準プロトコルのみを使い、独自実装しない。
- `OAuth 2.0` / `OIDC`:
- 公開アプリやネイティブアプリでは `Authorization Code + PKCE` を優先し、`Implicit` と `ROPC` は避ける。
- `state` と `nonce` を検証し、redirect URI は完全一致で検証し、open redirect を防ぐ。
- トークンは audience と scope に制約し、可能なら `DPoP` または `mTLS` で sender-constraining を行う。
- refresh token をローテーションし、ログアウトやリスクシグナル時に失効させる。
- `SAML`:
- `TLS 1.2+` を使い、response と assertion に署名し、機微 assertion は暗号化する。
- issuer、`InResponseTo`、タイムスタンプ (`NotBefore` / `NotOnOrAfter`)、`Recipient` を検証し、信頼済み鍵で検証する。
- 厳格なスキーマ検証と hardened XPath selection により XML signature wrapping を防ぐ。
- response の寿命を短く保ち、SP 起点フローを優先し、`RelayState` を検証し、replay 検出を実装する。

## トークン JWT と opaque token

- 単純さと失効容易性のため、サーバー管理型の opaque token を優先する。JWT を使う場合:
- アルゴリズムを明示的に固定し、`none` を拒否し、`iss` / `aud` / `exp` / `iat` / `nbf` を検証し、短寿命とローテーションを採用する。
- 秘密情報と鍵は `KMS` / `HSM` に安全に保管する。強い `HMAC` 秘密鍵または非対称鍵を使い、ハードコードしない。
- リプレイを減らすため、トークンをクライアント文脈 例: Cookie 内の fingerprint hash に結び付けることを検討する。
- ログアウトや重大イベント時の失効のため、denylist / allowlist を実装する。

## リカバリとリセット

- 既存アカウントと非存在アカウントに同じ応答を返し、列挙を防ぐ。処理時間も正規化する。
- 32 バイト以上の CSPRNG トークンを生成し、単回使用にし、ハッシュとして保存し、有効期限は短くする。
- リセットリンクは信頼済み固定ドメイン上の `HTTPS` を用い、UI では `Referrer-Policy: no-referrer` を追加する。
- リセット後は再認証を要求し、セッションをローテーションし、自動ログインしない。
- リセット試行だけでアカウントをロックしない。代わりにレート制限と監視を行う。

## 管理者アカウントと内部アカウント

- 管理者ログインは公開フォームと分離し、より強い MFA、端末健全性確認、IP allow-list、step-up 認証を強制する。
- 管理操作には別個のセッション文脈とより厳しいタイムアウトを用いる。

## 監視とシグナル

- 認証イベント 失敗 / 成功、MFA 登録 / 検証、リセット、ロックアウト を安定したフィールドと相関 ID 付きで記録し、秘密情報や生トークンは記録しない。
- credential stuffing を検知する。例: 高い失敗率、多数の IP / user agent、ありえない移動。新端末ログインはユーザーへ通知する。

## 実装チェックリスト

- パスワード: `Argon2id` 推奨 とユーザーごとの salt、定数時間検証、変更 / 設定時の漏えいパスワードチェック。
- MFA: 高リスク用途では `WebAuthn/passkeys` またはハードウェアトークン。フォールバックに `TOTP`。安全なバックアップコードによるリカバリ。
- フェデレーション: `Authorization Code + PKCE`、厳格な redirect URI 検証、audience / scope 強制、トークンローテーション。
- トークン: 短寿命、可能なら sender-constrained、失効機能あり、秘密情報は `KMS/HSM` 管理。
- リカバリ: 単回使用でハッシュ保存された期限付きトークン、一貫した応答、リセット後の再認証、セッションローテーション。
- 悪用対策: 認証エンドポイントにレート制限、スロットリング、異常検知、一貫したエラー処理。
- 管理者: 分離されたフロー、より厳しいポリシー、端末チェック。

## テスト計画

- ログイン、MFA 登録 / 検証、リセット、ロックアウトを対象に、均一なエラー応答を含む unit / integration test を作成する。
- `PKCE`、`state` / `nonce`、redirect URI 検証、token audience / scope のプロトコルテストを行う。
- credential stuffing 耐性と token replay に対する動的テストを行い、ログアウト後やロール変更後の失効を検証する。
