# MCP（Model Context Protocol）セキュリティガイドライン

- `rule_id`: `codeguard-0-mcp-security`
- 原題: `MCP (Model Context Protocol) Security Guidelines`

適切なセキュリティコントロールを実装せずに MCP サーバーまたはクライアントをデプロイしてはならない。

### ワークロードアイデンティティと認証
- 暗号化されたワークロードアイデンティティに SPIFFE／SPIRE を用いる。
  - SPIFFE（Secure Production Identity Framework For Everyone）はサービス識別の標準を提供する。
  - SPIRE（SPIFFE Runtime Environment）は短命の暗号アイデンティティ（SVID）を発行しローテーションする。

### 入力とデータのサニタイズ
- すべての信頼境界で許可リストを用いてすべての入力を検証する。
- ファイルパスは正規化（canonicalization）によってサニタイズする。
- データベース操作にはパラメータ化クエリを用いる。
- 文脈に応じた出力エンコーディング（SQL、シェル、HTML）を適用する。
- ツールの出力をサニタイズする：最小限のフィールドのみ返し、PII と機密データはすべてマスクする。
- すべての入力、ツールスキーマ、メタデータ、プロンプト、リソース内容を信頼できない入力として扱う。
- プロンプトインジェクション検知システムを導入する。
- 指示とデータの境界を維持するため、厳格な JSON スキーマを用いる。

### サンドボックスと分離
- MCP サーバーは最小権限で実行されるよう設計する。
- ホスト環境（ファイル、コマンド、ネットワーク）とやり取りする MCP サーバーは、サンドボックスコントロールを **実装しなければならない**。
- LLM が生成したコードは、ユーザーの完全な権限では **実行してはならない**。
- 追加のサンドボックス層を実装する：gVisor、Kata Containers、SELinux サンドボックスなど。

### リソースの暗号検証
- すべてのサーバーコードに暗号署名と SBOM を提供する。
- サーバーを読み込む前に MCP クライアントで署名検証を実装する。
- 転送中の **すべての** データに TLS を用いる。
- サーバーが期待するコードを実行していることを検証するリモートアテステーション機能を実装する。

### トランスポート層のセキュリティ

#### stdio トランスポート（ローカルサーバー）
- DNS リバインディングのリスクをなくすため、ローカル MCP には **強く推奨** する。
- パイプベースのストリーム通信。
- 権限昇格を防ぐサンドボックスを実装する。

#### HTTP ストリーミングトランスポート（リモートサーバー）
実装すべき必須のセキュリティコントロール：
- ペイロード上限（巨大ペイロードおよび再帰的ペイロードによる DoS の防止）
- ツール呼び出しおよびトランスポートリクエストのレート制限
- クライアントとサーバーの認証／認可
- 相互 TLS 認証
- TLS 暗号化
- CORS 保護
- CSRF 保護
- 整合性チェック（リプレイ、なりすまし、改ざんされたレスポンスの防止）

### 安全なツールと UX 設計
- 境界が明確な単一目的のツールを作る。「何でもできる」ツールは避ける。
- 検証や認可の判断を LLM に頼ってはならない。
- 影響の大きい操作には二段階コミットを用いる：第一段階でドラフト／プレビュー、第二段階で明示的な確認付きコミット。
- ロールバック／取り消しの経路（ドラフト ID、スナップショット、可逆な操作）と、可能なら時間制限付きコミットを提供する。

### ヒューマン・イン・ザ・ループ
- MCP サーバーでリスクの高い操作に確認プロンプトを実装する。
- リスクの高い操作について、MCP サーバー側でユーザー確認を求めるエリシテーションを用いる。
- セキュリティに関わるメッセージは、影響を **明確に** 示さなければならない。
- 人間の承認だけに頼ってはならない（ユーザーは疲労し得る）。

### ロギングと可観測性
- MCP サーバーとクライアントにロギングを実装する。
- ログに含める：使用されたツール、パラメータ、由来となったプロンプト。
- アクションのエンドツーエンドな追跡可能性のために OpenTelemetry を用いる。
- 操作と認可の不変レコードを維持する。

## デプロイパターンのセキュリティ

### すべてローカル（stdio または http）
- セキュリティはホストシステムの態勢に完全に依存する。
- DNS リバインディングのリスクを避けるため `stdio` トランスポートを用いる。
- 権限昇格攻撃を制限するためサンドボックスを用いる。
- 開発および個人利用に適する。

### シングルテナントリモート（http）
- クライアントとサーバー間の認証が **必須** である。
- 安全なクレデンシャル保管（OS キーチェーン、シークレットマネージャー）を用いる。
- 通信は **認証され暗号化されなければならない**。
- エンタープライズクライアントは、明示的な許可リストによる認証済みサーバー探索を強制すべきである。

### マルチテナントリモート（http）
- 堅牢なテナント分離、アイデンティティ、アクセス制御が必要である。
- 強力なマルチテナンシーコントロール（テナントごとの暗号化、ロールベースのアクセス制御）を実装する。
- サービスプロバイダーが直接ホストする MCP サーバーを優先する。
- 可能ならリモートアテステーションを提供する。
