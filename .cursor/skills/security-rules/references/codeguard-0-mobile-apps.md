# モバイルアプリケーションセキュリティガイドライン

- `rule_id`: `codeguard-0-mobile-apps`
- 原題: `Mobile Application Security Guidelines`

iOS および Android プラットフォーム向けに、安全なモバイルアプリを開発するための必須のセキュリティ実践。

### アーキテクチャと設計

最初から安全な設計原則を実装する：
- 最小権限と多層防御の原則に従う。
- 標準的で安全な認証プロトコル（OAuth2、JWT）を用いる。
- すべての認証および認可のチェックはサーバー側で行う。
- アプリとバックエンドサービスに必要な権限のみを要求する。
- アプリの更新、パッチ、リリースに対するセキュリティコントロールを確立する。
- 信頼でき検証済みのサードパーティライブラリおよびコンポーネントのみを用いる。

### 認証と認可

セキュリティ上の判断をクライアントに任せてはならない：
- 認証／認可はサーバー側のみで行う。
- ユーザーパスワードを端末に保存しない。失効可能なアクセストークンを用いる。
- モバイルアプリにクレデンシャルをハードコードしない。
- クレデンシャルは転送時に暗号化する。
- プラットフォーム固有の安全なストレージ（iOS Keychain、Android Keystore）を用いる。
- パスワードの複雑さを要求し、短い PIN（4 桁）を避ける。
- セッションタイムアウトとリモートログアウト機能を実装する。
- 機密操作には再認証を要求する。
- プラットフォームがサポートする生体認証を、安全なフォールバックとともに用いる。

### データの保管とプライバシー

保存時および転送時に機密データを保護する：
- プラットフォーム API を用いて機密データを暗号化する。独自暗号は避ける。
- ハードウェアベースのセキュリティ機能（Secure Enclave、Strongbox）を活用する。
- プライベートデータは端末の内部ストレージのみに保存する。
- PII の収集は必要最小限とし、自動失効を実装する。
- 機密データのキャッシュ、ログ、バックグラウンドのスナップショットを避ける。
- ネットワーク通信には常に HTTPS を用いる。

### ネットワーク通信

すべてのネットワーク通信は不安全と仮定する：
- すべてのネットワーク通信に HTTPS を用いる。
- 自己署名証明書のために SSL 証明書の検証を上書きしない。
- 適切な鍵長を伴う、強力で業界標準の暗号スイートを用いる。
- 信頼できる CA プロバイダーが署名した証明書を用いる。
- 追加のセキュリティとして証明書ピニングを検討する。
- SSL 経由でもデータを暗号化して送る。
- 機密データを SMS で送らない。

### コード品質と整合性

開発全体を通じてアプリケーションのセキュリティを維持する：
- 静的解析ツールで脆弱性を特定する。
- コードレビューでセキュリティを重視する。
- 既知の脆弱性に対処するため、すべてのライブラリを最新に保つ。
- 本番ビルドではデバッグを無効にする。
- アプリケーションコードの整合性を検証するコードを含める。
- アプリバイナリを難読化する。
- ランタイムの改ざん対策を実装する：
  - デバッグ、フッキング、コードインジェクションの検出
  - エミュレータまたは root／脱獄端末の検出
  - 実行時のアプリ署名の検証

### プラットフォーム固有のセキュリティ

#### Android セキュリティ
- コード難読化に Android の ProGuard を用いる。
- SharedPreferences に機密データを保存しない。
- バックアップに機密データが含まれるのを防ぐためバックアップモードを無効にする。
- ハードウェアバック（TEE または StrongBox）付き Android Keystore を用いる。
- 端末とアプリの整合性チェックに Google の Play Integrity API を実装する。

#### iOS セキュリティ
- 機密操作には Shortcuts の権限で端末のロック解除を要求するよう構成する。
- 機密機能について Siri インテントの `requiresUserAuthentication` を true にする。
- ディープリンクのエンドポイントで認証チェックを実装する。
- ロック画面では機密ウィジェット内容をマスクする条件分岐を用いる。
- 機密データは plist ではなく iOS Keychain に保存する。
- 暗号鍵の保管に Secure Enclave を用いる。
- アプリ整合性の検証に App Attest API を実装する。
- 永続的な端末状態の追跡に DeviceCheck API を用いる。

### テストと監視

包括的なテストでセキュリティコントロールを検証する：
- 暗号関連の脆弱性評価を含むペネトレーションテストを実施する。
- セキュリティ機能が意図どおり動作することを自動テストで確認する。
- セキュリティ機能が使いやすさを損なわないことを確認する。
- 脅威を検知し対応するためリアルタイム監視を用いる。
- 明確なインシデント対応計画を用意する。
- 定期的な更新を計画し、必要に応じて強制更新メカニズムを実装する。

### 入力と出力の検証

インジェクションと実行攻撃を防ぐ：
- すべてのユーザー入力を検証しサニタイズする。
- インジェクション攻撃を防ぐため出力を検証しサニタイズする。
- ショルダーハッキングを防ぐため UI フィールドで機密情報をマスクする。
- セキュリティ関連の活動（新しい端末からのログインなど）をユーザーに通知する。

OWASP Mobile Application Security フレームワークに基づくこれらの実践に従うことで、開発および運用の両フェーズにおいて、モバイルアプリケーションのセキュリティ態勢を大きく改善できる。
