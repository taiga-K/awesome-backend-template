# ファイルアップロードのセキュリティガイドライン

- `rule_id`: `codeguard-0-file-handling-and-uploads`
- 原題: `File Upload Security Guidelines`

本ルールは、悪意のあるファイル攻撃を防ぎシステムの整合性を守るための、安全なファイルアップロードの実践について述べる。

- 拡張子の検証
  - 許可する拡張子は、業務上重要な機能にのみリストする。
  - 拡張子を検証する前に入力検証を適用する。
  - 二重拡張子（例: `.jpg.php`）やヌルバイト注入（例: `.php%00.jpg`）を避ける。
  - 拡張子は拒否リストではなく許可リストで検証する。
  - ファイル名をデコードした後に拡張子を検証し、バイパスを防ぐ。

- コンテンツタイプとファイルシグネチャの検証
  - クライアントが送る Content-Type は偽装可能なので信頼しない。
  - Content-Type チェックと併せてファイルシグネチャ（マジックナンバー）を検証する。
  - MIME タイプは許可リストで、素早い防御層として使う。
  - ファイルシグネチャ検証は単独のセキュリティ対策としては使わない。

- ファイル名のセキュリティ
  - ユーザが指定した名前ではなく、ランダムなファイル名（UUID／GUID）を生成する。
  - ユーザファイル名が必要なら、最大長を制限する。
  - 文字は英数字、ハイフン、スペース、ピリオドのみに制限する。
  - 先頭のピリオド（隠しファイル）や連続ピリオド（ディレクトリトラバーサル）を防ぐ。
  - シェルスクリプト処理の安全のため、先頭のハイフンやスペースを避ける。

- ファイル内容の検証
  - 画像には、悪意のある内容を破壊する画像の書き換え技術を適用する。
  - Microsoft 文書には Apache POI で検証する。
  - ZIP は攻撃面が多いため避ける。
  - リソースが許せば、サンドボックス環境で手動レビューする。
  - 該当するファイル種別ではウイルス対策スキャンと Content Disarm & Reconstruct（CDR）を組み込む。

- ストレージのセキュリティ
  - 可能なら完全分離のため別サーバに保存する。
  - Web ルート外に保存し、管理アクセスのみとする。
  - Web ルートに置く場合は、適切なアクセス制御で書き込み専用にする。
  - 公開アクセスは ID からファイル名へマッピングするアプリハンドラを使う。
  - DBA の専門知識とともに、特定用途では DB 保存を検討する。

- アクセス制御と認証
  - ファイルアップロードの前にユーザ認証を必須とする。
  - ファイルの参照・変更に適切な認可レベルを実装する。
  - ファイルシステムの権限は最小権限の原則で設定する。
  - 実行権限が必要なら、実行前にファイルをスキャンする。

- アップロードとダウンロードの制限
  - アップロード保護のため適切なファイルサイズ上限を設定する。
  - 圧縮ファイルは展開後サイズの上限も検討する。
  - DoS を防ぐためダウンロードサービスにリクエスト上限を設ける。
  - ZIP ファイルのサイズを安全に計算する方法を使う。

- 追加のセキュリティ対策
  - ファイルアップロードのエンドポイントを CSRF から保護する。
  - ファイル処理に使うライブラリを安全に構成し更新する。
  - アップロード活動のログと監視を実装する。
  - 違法コンテンツのユーザ報告の仕組みを用意する。
  - 圧縮ファイルは安全な展開方法を使う。

要約:  
多層的な検証、安全な保存、適切なアクセス制御、包括的な監視により、ファイルアップロードに多層防御を実装する。単一の検証手法だけに頼らず、常に安全なファイル名を生成して攻撃を防ぐ。
