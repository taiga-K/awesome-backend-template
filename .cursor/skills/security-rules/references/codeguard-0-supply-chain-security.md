# 依存関係とサプライチェーンのセキュリティ

- `rule_id`: `codeguard-0-supply-chain-security`
- 原題: Dependency & supply chain security (pinning, SBOM, provenance, integrity, private registries)

エコシステム全体でサードパーティのリスクを管理する。選定とピン留めから出所、スキャン、迅速な対応までを含む。

### 方針とガバナンス
- 許可リスト化したレジストリとスコープを維持する。信頼できないソースからの直接インストールは禁止する。
- ロックファイルとバージョンのピン留めを必須とする。イメージやベンダー資産はダイジェストのピン留めを優先する。
- アプリ/イメージ用に SBOM を生成し、成果物とともに保管する。出所を証明する（SLSA、Sigstore）。

### パッケージの衛生（npm 中心だが他にも適用可能）
- 定期的に監査（`npm audit`、エコシステムの SCA）しパッチを当てる。深刻度に応じた SLA を強制する。
- 決定的ビルド: CI/CD では `npm install` ではなく `npm ci`。ロックファイルの一貫性を維持する。
- 可能ならインストール時に実行されるインストールスクリプトを避ける。リスクをレビューする。
- `.npmrc` でプライベートレジストリをスコープする。ワイルドカードレジストリを避け、整合性検証を有効にする。
- 公開用アカウントで 2FA を有効にする。

### 開発プラクティス
- 依存の足跡を最小化する。未使用パッケージを削除する。些末なタスクは標準ライブラリ/自社実装を優先する。
- タイポスクワッティングや protestware 対策: メンテナをピン留めし、リリースを監視し、出所チェックを使う。
- ヘルメティックビルド: 必要でない限りコンパイル/パッキング段階でネットワークを使わない。真正性チェック付きでキャッシュする。

### CI/CD 統合
- SCA、SAST、IaC スキャンをゲートに組み込む。クリティカルは失敗とする。上書きには補償的統制付きの承認を要求する。
- 成果物に署名し、デプロイ時に署名を検証する。アドミッションでポリシーを強制する。

### 脆弱性管理
- パッチ済み脆弱性: 更新をテストしてデプロイする。API の破壊的変更は文書化する。
- 未パッチ脆弱性: CVE の種類に基づき補償的統制（入力検証、ラッパーなど）を実装する。間接依存の回避策より直接依存の修正を優先する。
- リスク判断を文書化し、ビジネス上の根拠とともに適切な権限へエスカレーションする。

### インシデント対応
- 迅速なロールバックを維持する。侵害されたパッケージを隔離し、ロールアウトを抑制し、関係者に通知する。
- 脅威インテリフィード（例: npm アドバイザリ）を監視し、クリティカル CVE でチケットを自動起票する。

### 実装チェックリスト
- ロックファイルあり。整合性チェックオン。プライベートレジストリ設定済み。
- SBOM と出所を保管。デプロイ前に署名を検証。
- テストとレビューゲート付きの自動依存更新。
- 高深刻度の脆弱性は SLA 内で修正するか、緩和して文書化する。
