# 認可とアクセス制御

- `rule_id`: `codeguard-0-authorization-access-control`
- 原題: `Authorization & Access Control`

最小権限と精密なアクセス判断を、すべてのリクエストとリソースに対して適用する。`IDOR` と mass assignment を防ぎ、必要な場面では強力な取引認可を提供する。

## 基本原則

1. デフォルト拒否: すべてのアクセス要求の既定値は拒否とする。ロールやユーザーへは拒否ではなく明示的な許可を与える。許可ルールに一致しなければ `HTTP 403 Forbidden` を返す。
2. 最小権限の原則: ユーザーには職務遂行に必要な最小限の権限だけを付与する。権限が過剰になっていないかを定期的に監査する。
3. すべてのリクエストで権限を検証する: `AJAX`、API、直接アクセスなどソースを問わず、すべてのリクエストで認可を確認する。middleware や filter で一貫して強制する。
4. `RBAC` より `ABAC/ReBAC` を優先する: 単純なロールベース制御ではなく、属性ベースまたは関係ベースのアクセス制御で細粒度の権限を扱う。

## 体系的コントロール

- middleware、policy、filter により、サービス境界で認可を集中管理する。
- 権限はリソース単位 所有者、テナンシー でモデル化し、データクエリにもスコープ制約を適用する。
- リソース存在の漏えいを避けるため、汎用的な `403/404` を返す。
- すべての拒否を、ユーザー、操作、リソース識別子 非 PII、理由コード付きで記録する。

## IDOR 防止

- ユーザー提供 ID だけを信用しない。各オブジェクト実体へのアクセス権を必ず検証する。
- ユーザースコープ付きクエリまたはサーバー側ルックアップでリソースを解決する。例: `Project.find(id)` ではなく `currentUser.projects.find(id)`。
- 防御の重ね掛けとして、列挙困難な識別子 `UUID` やランダム ID を使う。ただし秘匿性のみに依存しない。

## Mass Assignment 防止

- リクエスト本文を、機微フィールドを含むドメインオブジェクトへ直接バインドしない。
- DTO で安全に編集可能なフィールドだけを公開し、patch / update には明示的 allow-list を維持する。
- allow-list が難しい場合でも、フレームワーク機能で機微フィールドを block-list する。

## 取引認可 Step-Up

- 送金、権限昇格、データエクスポートなど機微な操作には第 2 要素を要求する。`What-You-See-Is-What-You-Sign` を適用し、重要フィールドを確認画面に表示する。
- 各トランザクションごとに一意で短寿命な認可資格情報を使い、途中でデータが変わったら拒否する。
- 選択した認可方式をサーバー側で強制し、クライアント側ダウングレードを防ぐ。
- brute-force 対策としてスロットリングを行い、失敗時にはフロー全体をやり直させる。

## テストと自動化

- エンドポイント / リソース、ロール / 属性、期待結果を列挙した認可マトリクス `YAML/JSON` を維持する。
- マトリクスを反復し、ロールトークンを発行して allow / deny を検証する integration test を自動化する。トークン期限切れや失効ケースも含める。
- ID 入れ替え、ロール降格、scope 欠落、バイパス試行などの否定ケースを実施する。

## 実装チェックリスト

- middleware / policy により、デフォルト拒否と各エンドポイントのリソースチェックを強制する。
- クエリスコープにより、ユーザーが許可された行 / オブジェクトだけへアクセスできるようにする。
- DTO と allow-list によって mass assignment を防ぎ、機微フィールドはバインド不能にする。
- 機微操作では、一意で短寿命な資格情報による step-up 認可を導入する。
- 認可マトリクスを CI テストの入力にし、失敗時はマージをブロックする。
