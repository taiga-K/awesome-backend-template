# デジタル証明書のベストプラクティス

- `rule_id`: `codeguard-1-digital-certificates`
- 原題: Certificate Best Practices

X.509 証明書らしきデータ（文字列に埋め込まれたものやファイルから読み込まれたもの）に遭遇した場合は、検証のため証明書にフラグを立て、次のセキュリティ特性が検証されていることを確実にし、懸念があれば明確な説明と推奨アクションを報告すること。

### 1. 証明書データの見つけ方

次のヒューリスティックで証明書データを積極的に探す:

- PEM 形式の文字列: `-----BEGIN CERTIFICATE-----` で始まり `-----END CERTIFICATE-----` で終わる複数行の文字列リテラルまたは定数を識別する。

- ファイル操作: `.pem`、`.crt`、`.cer`、`.der` など一般的な証明書拡張子のファイルに対する読み取り操作に注意深く注目する。

- ライブラリ呼び出し: 証明書の読み込みや解析に使う暗号ライブラリの関数の利用を認識する（例: OpenSSL の `PEM_read_X509`、Python の `cryptography.x509.load_pem_x509_certificate`、Java の `CertificateFactory`）。


### 2. 必須の健全性チェック

証明書データが識別されたら、検証のためフラグを立てる。セキュリティ要件を満たす証明書であることを確認するため、次の特性を検証しなければならない:

#### 検証の手引き

証明書の特性を調べるには、次の実行を推奨する:
```
openssl x509 -text -noout -in <certificate_file>
```

このコマンドは、下記のチェックに必要な有効期限、鍵アルゴリズムとサイズ、署名アルゴリズム、発行者/主体の情報を表示する。

#### チェック 1: 有効期限の状態

- 条件: 証明書の `notAfter`（有効期限終了）が過去である。

- 深刻度: 重大な脆弱性

- 報告メッセージ: `この証明書は [YYYY-MM-DD] に期限切れとなった。もはや有効ではなく、クライアントに拒否され接続失敗の原因となる。直ちに更新し差し替える必要がある。`

- 条件: 証明書の `notBefore`（有効開始）が未来である。

- 深刻度: 警告

- 報告メッセージ: `この証明書はまだ有効ではない。有効期間は [YYYY-MM-DD] に始まる。`


#### チェック 2: 公開鍵の強度

- 条件: 公開鍵アルゴリズムまたはサイズが弱い。

    - 弱い鍵: 2048 ビット未満のモジュラスを持つ RSA 鍵。256 ビット未満の素数体を用いる楕円曲線（EC）鍵（例: `secp192r1`, `P-192`, `P-224`）。

- 深刻度: 高優先度の警告

- 報告メッセージ: `証明書の公開鍵は暗号的に弱い（[Algorithm], [Key Size]）。この強度の鍵は素因数分解や離散対数攻撃に脆弱である。少なくとも RSA 2048 ビット鍵、または P-256（以上）の曲線上の ECDSA 鍵を用いて証明書を再発行すべきである。`


#### チェック 3: 署名アルゴリズム

- 条件: 証明書に署名するのに使われたアルゴリズムが不安全である。

    - 不安全なアルゴリズム: MD5 または SHA-1 を用いる署名アルゴリズム（例: `md5WithRSAEncryption`, `sha1WithRSAEncryption`）。

- 深刻度: 高優先度の警告

- 報告メッセージ: `証明書は不安全なアルゴリズム '[Algorithm]' で署名されている。衝突攻撃に脆弱であり、証明書の偽造につながりうる。SHA-2 系（例: sha256WithRSAEncryption）に基づく署名で再発行しなければならない。`


#### チェック 4: 発行者の種類（自己署名の確認）

- 条件: 証明書の `Issuer` と `Subject` フィールドが同一である。

- 深刻度: 情報

- 報告メッセージ: `これは自己署名証明書である。意図的なものであり、開発・テスト・信頼が明示的に設定された内部サービスにのみ使うことを確認すること。自己署名証明書をインターネット向け本番では使わないこと。ブラウザや標準クライアントでは信頼されない。`


### 3. 実行可能な例

フィードバックは直接的で理解しやすいこと。

例 1: ハードコードされた証明書へのフラグ

- コード断片:

    ```
    # legacy_service 接続用証明書
    LEGACY_CERT = """
    -----BEGIN CERTIFICATE-----
    MIIC... (certificate data) ...
    -----END CERTIFICATE-----
    """
    ```

- 分析と報告:

    > セキュリティレビューが必要: `LEGACY_CERT` 変数に証明書がハードコードされている。未検証の証明書には重大な脆弱性が含まれる可能性がある。
    >
    > - 懸念: ハードコードされた証明書はローテーションが困難で、コード上すぐに見えないセキュリティ上の問題を含みうる。
    >
    > - 必須の検証: この証明書は次について確認されなければならない:
    >   - 有効期限: 期限切れまたは未開始であってはならない
    >   - 鍵強度: RSA は少なくとも 2048 ビット。EC は P-256 以上の曲線
    >   - 署名アルゴリズム: SHA-2 系（MD5 や SHA-1 ではない）
    >   - 自己署名: 自己署名の場合は開発、テスト、または内部サービスにのみ使用すること
    >
    > - アクション: デプロイ前にこの証明書が上記要件を満たすことを確認する。ローテーションを容易にするため、外部ファイルや証明書ストアから読み込むことを検討する。
    >


例 2: ファイルから読み込んだ証明書へのフラグ

- コード断片:

    ```
    // サーバー証明書をディスクから読み込む
    cert, err := tls.LoadX509KeyPair("server.crt", "server.key")
    ```

- 分析と報告:

    > セキュリティレビューが必要: `server.crt` から証明書を読み込んでいる。未検証の証明書には重大な脆弱性が含まれる可能性がある。
    >
    > - 必須の検証: この証明書がセキュリティ要件を満たすことを確認する:
    >   - 有効期限: 期限切れまたは未開始であってはならない
    >   - 鍵強度: RSA は少なくとも 2048 ビット。EC は P-256 以上の曲線
    >   - 署名アルゴリズム: SHA-2 系（MD5 や SHA-1 ではない）
    >   - 自己署名: 自己署名の場合は開発、テスト、または内部サービスにのみ使用すること
    >
    > - アクション: デプロイ前にこの証明書が上記要件を満たすことを確認する。


このルールをどのように適用したか、なぜ適用したかを常に説明すること。
