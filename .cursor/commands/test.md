# test

Go の単体テストを実行する。基本は CI と同じ `go test ./...` をリポジトリルートで実行し、失敗時は落ちたパッケージ・テスト名・主な原因を要約する。

## いつ使うか

- ユーザーが「単体テストを実行して」「テストを回して」「`go test` を実行して」と依頼したとき
- 実装やリファクタリングのあと、Go の変更が既存挙動を壊していないか確認したいとき
- commit や PR の前に、ローカルで Go のテスト結果を確認したいとき

## テスト実行ルールの参照先

- 開発方針は `CONTRIBUTING.md` に従う
- `.cursor/commands/test.md` では、Go の単体テストを安全に実行する手順のみを扱う

## 事前確認

### 1. 追跡対象の Go ファイルを確認

```bash
git ls-files "*.go"
```

- 追跡対象の Go ファイルが 1 つもなければ、`go test` は実行せず「テスト対象なし」として報告する

### 2. `go.mod` の有無を確認

```bash
ls go.mod
```

- `go.mod` がなければ `go test` は実行せず、「Go ファイルはあるが `go.mod` がないためスキップ」と報告する
- これは CI の `.github/workflows/go-static-analysis.yml` と同じ判断に合わせる

## 基本フロー

### 1. リポジトリルートで単体テストを実行

```bash
go test ./...
```

### 2. 結果を確認

- 成功した場合は、`go test ./...` が成功したことを報告する
- 失敗した場合は、失敗したパッケージ、テスト名、主要なエラーメッセージを要約して報告する
- 失敗ログが長い場合は、原因特定に必要な箇所を優先して要約する

## よく使う実行例

**特定パッケージ配下だけ確認したい場合:**

```bash
go test ./internal/...
```

**キャッシュを使わず再実行したい場合:**

```bash
go test -count=1 ./...
```

**競合やデータレースが気になる場合:**

```bash
go test -race ./...
```

## 補足

- 既定の実行コマンドは `go test ./...` とし、追加オプションはユーザーが明示した場合や必要性が高い場合のみ使う
- 失敗時に自動で修正を始めず、まずテスト結果を報告して次の対応方針を確認する
- commit 前や PR 前の確認では、CI と同じ `go test ./...` を優先する
