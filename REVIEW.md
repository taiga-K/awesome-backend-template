# コードレビュアーガイドライン

最終更新: 2026/4/14

## 基本情報

本ガイドラインは、本プロジェクトに対するコードレビューの判断基準を定義するものです。対象読者は実務エンジニアであり、レビュー時にそのまま使える実践的な内容に絞っています。

本プロジェクトは Go を前提とした `DDD x クリーンアーキテクチャ` のバックエンドテンプレートです。レビューでは一般的な Go の品質基準に加え、`cmd/api`、`internal/domain`、`internal/usecase`、`internal/presentation`、`internal/infra`、`docs/domain`、`docs/api` の責務境界を守れているかを重視します。

現時点では個人開発を起点としつつ、将来的な小規模チーム開発でも破綻しない運用を想定します。そのため、レビューでは「今動くか」だけでなく、「今後の変更で壊れにくいか」「次の開発者が迷わないか」まで確認します。

## 1. レビューの基準・厳格さ

#### コードレビューの基本方針

本プロジェクトのレビューは、品質ゲートとして機能させます。明確な改善であり、アーキテクチャ境界、正確性、テスト、ドキュメント整合性に問題がなければ承認します。一方で、将来の変更を難しくする設計崩れや、レイヤー違反を含む変更は、たとえ動作していても差し戻します。

#### 承認の判断基準

以下をすべて満たす場合に承認します。

- 変更の責務が明確であり、既存コードベースの理解しやすさと保守性を下げていない
- 機能または修正内容が期待どおりに実装されている
- `domain`、`usecase`、`presentation`、`infra` の境界を壊していない
- 変更内容に応じたテストまたは妥当な検証結果がある
- 関連するドキュメントや API 契約との差分が整理されている
- セキュリティ、エラーハンドリング、運用上のリスクが放置されていない

#### Nitコメントの扱い

`Nit` は改善提案として扱い、原則として承認ブロックには使いません。ただし、同種の `Nit` が積み重なって可読性や一貫性を崩す場合は、単なる好みではなく保守性の問題として通常指摘へ引き上げます。

## 2. レビュー速度の目標

#### レビューリクエストへの初回応答時間

原則として 1 営業日以内に初回応答します。すぐに詳細レビューできない場合でも、受領したこと、確認予定時刻、先に見てほしい観点の有無を返してください。

#### レビューの優先順位付け

レビュー優先度は、利用者影響と開発の詰まり具合で判断します。本番障害対応、セキュリティ修正、他作業をブロックしている PR を優先し、通常の改善や内部整理はその次に扱います。

#### 集中したタスクの最中でのレビューの扱い

集中作業中でも、緊急性が高いレビューは優先します。緊急でない場合は作業の区切りで確認し、レビュー待ちが長時間放置されないようにします。

## 3. CLサイズの基準

#### 理想的なCLのサイズ

理想は、1 つの意図を 1 つの Pull Request に閉じ込めた差分です。レビューコメントが責務ごとに整理でき、変更理由と影響範囲を追いやすい大きさを保ってください。

#### 「大きすぎる」と判断するCLのサイズ

目安として、レビュー対象が 800 行を超えたら大きめ、1000 行を超えたら注意が必要と判断します。行数だけでなく、関心事が複数混ざっていてレビューの観点が散る場合も、大きすぎる PR と見なします。

#### 大きなCLへの対応方針

大きな PR では、レビュアーは影響範囲、主要な設計判断、テストの妥当性、見落としやすい副作用を通常より慎重に確認します。投稿側は Pull Request の説明文で変更の背景、主な論点、見てほしい観点を明確にしてください。

## 4. 確認項目の優先順位

#### レビューで確認する項目の優先順位

本プロジェクトでは、次の順番で確認します。

1. アーキテクチャ境界と責務の置き場所
2. 機能性、正確性、失敗時の振る舞い
3. テストの妥当性と再発防止力
4. 複雑さ、可読性、変更しやすさ
5. Go らしい API 設計、命名、エラー処理
6. ドキュメントと外部契約の整合性
7. スタイル、コメント、細かな表記ゆれ

#### すべての行をレビューする厳格さ

差分の全行を必ず確認します。ただし、同じ重要度で見るのではなく、まず重大な設計問題や挙動の誤りを潰し、その後に可読性や表現の改善を行います。

#### ディレクトリごとの重点観点

差分の主戦場に応じて、重点観点を切り替えます。

- `internal/domain`: 業務ルール、不変条件、用語の妥当性、ORM や HTTP 都合の混入有無を確認します
- `internal/usecase`: 調停責務、トランザクション境界、副作用順序、DTO の境界を確認します
- `internal/presentation`: request/response 変換、入力検証、認証情報、エラー表現の変換を確認します
- `internal/infra`: repository 実装、外部クライアント、ORM/SQL/SDK の閉じ込め、再構成、排他制御を確認します
- `docs/domain`: 用語、ユースケース、モデルの説明がコード変更と一致しているかを確認します
- `docs/api`: OpenAPI と実装・ユースケースの整合性、分割構成の崩れがないかを確認します
- `cmd/api` と `internal/config`: 起動配線、依存注入、設定解釈が本体ロジックを抱え込んでいないかを確認します

## 5. テスト要件

#### テストの必須度

バグ修正、新しい業務ロジック、外部契約変更、失敗条件や境界条件に関わる修正では、原則としてテストが必要です。文書のみの更新、コメント修正、純粋なリネームや整形のように挙動を変えない変更では、テスト追加を必須にしません。

#### 必要なテストの種類

変更内容に応じて、以下を使い分けます。

- ドメインやユースケースの判断ロジックにはユニットテストを追加します
- `presentation` と `infra` では、変換や接続境界の誤りを拾えるテストを優先します
- バグ修正では、再発防止になる失敗ケースのテストを追加します
- API 契約や外部連携に影響する変更では、少なくとも主要経路の統合的な検証方法を PR に明記します

#### テストカバレッジの目標

固定の数値目標だけで判断しません。重要なのは、仕様の境界、失敗系、再発防止点を押さえたテストになっていることです。薄いテストを大量に足して見かけのカバレッジだけを上げる変更は評価しません。

#### レビューで見るテスト品質

以下を満たしているかを確認します。

- 仕様の境界条件と主要な失敗パターンを押さえている
- テスト名と失敗メッセージから意図が読み取れる
- 実装詳細の固定ではなく、振る舞いの保証になっている
- 重複が多い場合は table-driven test などで整理されている

## 6. スタイルガイド

#### 使用するスタイルガイド

本プロジェクトでは、一般的な Go の慣用に加えて、`DDD x クリーンアーキテクチャ` と高凝集・低結合の方針を独自スタイルガイドとして採用します。見た目の整形よりも、責務の置き方、依存方向、命名、ゼロ値安全性、エラー設計を優先します。

#### スタイルガイドへの準拠

準拠は必須です。特に、`common`、`util`、`shared` のような曖昧な退避先を増やす変更、内側の層へ外側の都合を漏らす変更、`domain` に ORM や HTTP の都合を持ち込む変更は、スタイル違反ではなく設計違反として扱います。

#### 自動フォーマッターの使用について

Go のコードは `gofmt` を前提とします。OpenAPI や関連設定も既存の構成に従い、一貫した形式で管理してください。フォーマットされていないだけの差分でレビュー時間を使わないようにします。

## 7. コメントのラベル体系

#### 使用するコメントラベル

レビューコメントでは、意図が一目で分かるように必要に応じてラベルを付けます。

- `Critical`: 重大な欠陥、障害、重大な設計破綻
- `Blocking`: 承認前に必ず修正が必要な問題
- `Security`: 認証、認可、情報漏えい、入力検証などの安全性問題
- `Performance`: 明確な性能リスクや無駄なコスト増加
- `Nit`: 軽微な改善提案
- `Optional`: 任意の改善案
- `FYI`: 背景共有や参考情報

## 8. コメント要件

#### コードコメントの言語

コードコメントは日本語を基本とします。外部仕様や標準用語で英語を使う必要がある場合でも、読者が判断を誤らないことを優先してください。

#### コードコメントの必須度

複雑な分岐、失敗方針、並行処理、外部制約との橋渡しなど、コードだけでは意図が読み切れない箇所にはコメントが必要です。一方で、処理をそのまま言い換えるだけのコメントは不要です。

#### 不明点を説明するコメント

自明でない設計判断は、なぜそうしているかを残してください。特に、トランザクション順序、再試行方針、値オブジェクト化しない理由、外部都合をどこで吸収しているかは、後から見て分かるようにします。

## 9. ドキュメント要件

#### ドキュメント更新の必須度

変更によって利用方法、挙動、用語、API 契約、ユースケースの理解に影響がある場合は、関連ドキュメントの更新を必須とします。`CONTRIBUTING.md` の方針どおり、`README.md` に影響する変更では必ず更新してください。

#### 更新対象の判断基準

どの文書を更新するかは、変更の性質で判断します。

- 業務用語や意味の変更は `docs/domain/ubiquitous-language.md` を確認します
- ユースケースの流れが変わる場合は `docs/domain/usecase` を更新対象に含めます
- API 契約の変更は `docs/api` を更新対象に含めます
- セットアップや利用方法の変更は `README.md` を更新対象に含めます

## 10. 専門領域のレビュアー

#### 専門レビューの必須度

通常の変更では必須ではありませんが、境界をまたぐ変更や判断が重い変更では、専門観点での確認を強く推奨します。特に、`domain` の用語変更、トランザクション設計、永続化戦略、OpenAPI 変更は、関連領域に詳しいレビューが有効です。

#### 専門レビューが必要になりやすい変更

以下のような変更では、追加レビューを検討してください。

- `domain` の概念や不変条件を変更するもの
- `usecase` で副作用順序やトランザクション境界を変えるもの
- `infra` で repository 実装や外部 API 接続方式を変えるもの
- `docs/api` を含む外部契約変更

## 11. 緊急時の定義

#### 「緊急」と見なす状況

本番障害の修正、セキュリティホールの是正、利用不能またはデータ破損につながる重大不具合への対応を緊急と見なします。

#### 緊急時のレビュープロセス

緊急時でもレビューは省略しません。通常より短いサイクルで確認し、最低限として修正の妥当性、影響範囲、ロールバック方針、追加で必要な追跡作業を明記してください。

#### 緊急対応後の徹底的なレビュー

事後レビューは必須です。応急処置で入れた分岐、暫定実装、監視不足、文書未更新を後続 PR で放置しないようにします。

## 12. 対立解決

#### レビュアーと開発者の意見が対立した場合のエスカレーション先

まずは Pull Request 上で、現象、懸念、代替案、採用基準を明確にします。それでも合意できない場合は、関係者で方針をそろえ、必要なら文書へ残します。

#### 対立解決のための対面会議

コメントだけで長引く場合は、短時間の同期会話で解決して構いません。ただし、結論は必ず Pull Request か関連文書に戻し、経緯が追えない状態を作らないでください。

#### 対立解決の期限

通常は 3 営業日以内を目安に結論を出します。緊急対応やブロッカーになっている場合は、当日中に判断経路を決めます。

## 13. LGTM with Comments

#### LGTM with Commentsを使用する条件

軽微な表現修正、命名調整、コメント追記、テスト説明の補足など、修正の方向が明確でリスクが低い場合に使用します。アーキテクチャ境界、失敗時挙動、データ整合性、公開契約に関わる論点では使いません。

#### 「開発者が適切に対処すると確信できる」基準

修正内容が具体的で解釈の余地が少なく、これまでのやり取りから認識齟齬がないと判断できることを基準にします。人ではなく、変更内容の明確さで判断してください。

## 14. 「後でクリーンアップ」の許容

#### 「後でクリーンアップします」という約束の扱い

原則として、その PR で必要な品質はその PR で満たします。後で直すことを前提に設計崩れやテスト不足を見逃してはいけません。例外は、緊急修正で切り戻しや被害抑止を最優先にする場合だけです。

#### 例外を認める場合の条件

例外を認める場合は、スコープ、残課題、期限、追跡方法を Pull Request に明記してください。曖昧な「後でやる」は許容しません。

## 15. メンタリング

#### コードレビューを学習機会としてどれくらい重視するか

品質確保を主目的としつつ、将来同じ迷いを減らせるレビューを重視します。特にこのリポジトリでは、Go の慣用と `DDD x クリーンアーキテクチャ` の責務分離を学べるコメントが価値を持ちます。

#### 教育的なコメント

教育的なコメントは歓迎しますが、抽象論で終わらせず、差分上の現象、問題になる理由、直し方の方向性まで書いてください。レビューコメントは設計メモではなく、意思決定を前に進めるものとして扱います。

#### ジュニア開発者に対する特別な配慮

品質基準は変えません。ただし、背景知識が必要な指摘では、なぜその層に置かないのか、なぜそのエラー設計にするのかを短く補足し、再現可能な学びにしてください。

## 16. AIコードレビュー

#### AIコードレビューツールを利用するか

積極的に活用して構いません。

#### AIレビューツールのカスタマイズ

ルールが設定できる場合は、少なくとも以下を反映してください。

- `domain` に ORM/HTTP 詳細を持ち込まない
- `usecase` を巨大な手続き置き場にしない
- `presentation` に業務ロジック本体を置かない
- `infra` から外部 SDK の詳細を内側へ漏らさない
- ドキュメント更新漏れを拾う

## 17. API変更の確認

#### API変更がある場合における、実際の動作確認

本プロジェクトはバックエンド API 開発を前提とするため、レビューでは UI の有無ではなく API の契約と挙動を確認します。API 入出力やクライアント影響がある変更では、コード差分だけで済ませず、少なくともリクエストとレスポンスの実測、または再現可能な確認手順を確認してください。

#### 専門レビューの必要性

通常は UI や UX の専門レビューは不要です。代わりに、API 利用者が困らない入出力契約、エラーレスポンス、認証認可、後方互換性の観点でレビューします。OpenAPI を更新する変更では、実装と `docs/api` の整合性も合わせて確認してください。

## 18. ポジティブフィードバック

#### 良いコードや改善に対するポジティブなコメントの方針

良い分割、明快なテスト、境界を守った設計、丁寧な文書更新は積極的に言語化してください。何が良かったかを具体的に残すと、チームの基準が揃いやすくなります。

#### ポジティブフィードバックの頻度

毎回無理に書く必要はありませんが、再利用したい実装や判断があれば短く残します。称賛そのものより、再現できる学びになるコメントを優先してください。

## 19. 継続的改善

#### ガイドラインを見直す頻度

四半期ごと、または大きな設計方針の変更時に見直します。`DDD x クリーンアーキテクチャ` の実運用で詰まりやすかった点が見えたら、その都度反映して構いません。

#### ガイドラインへのフィードバック収集方法

Pull Request や Issue で具体的な差分と一緒に提案してください。抽象的な感想より、「どのレビューで困ったか」「何を判断できなかったか」を残すほうが改善につながります。

## 20. メトリクスと測定

#### 追跡するレビューメトリクス

最低限、以下を継続的に見ます。

- 初回応答までの時間
- レビュー完了までの時間
- レビュー後に発覚した不具合件数
- 差し戻し理由の傾向

#### メトリクスの使用目的

目的は個人評価ではなく、レビュー運用の改善です。たとえば、差し戻し理由に「レイヤー違反」や「ドキュメント更新漏れ」が多いなら、ガイドラインやテンプレート側を見直します。

## 付録: レビュー時の実務チェックポイント

差分を読むときは、まず「この変更はどの層の責務か」を確認してください。`domain` にビジネスルールがあり、`usecase` が調停し、`presentation` が外部入出力をさばき、`infra` が実装詳細を閉じ込める、という前提から外れていないかを最初に見ます。

次に、正常系だけでなく失敗時の動き、エラーの返し方、再試行の可否、テストで固定されている範囲を確認します。Go のレビューでは、命名、早期 return、エラーラップ、ゼロ値安全性、`interface` の作り過ぎ、隠れた副作用、並行処理の終了条件を重点的に見てください。

最後に、変更によって説明責任が増えていないかを確認します。利用方法が変わるなら `README.md`、業務上の意味が変わるなら `docs/domain`、外部契約が変わるなら `docs/api` を更新対象に含め、コードだけが先行しない状態を保ってください。
